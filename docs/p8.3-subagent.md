# Phase 8.3 ‚Äî Subagent / Task ‚Äî design PRD

> Status: design, ready to build. The spec to hand a developer for the **first
> slice** of subagent delegation. Companion to the Phase 8 roadmap in
> [README.md](../README.md) and to the shipped MCP adapter (8.2,
> [internal/mcp](../internal/mcp)). Like P7, this phase is mostly a *consumer* of
> seams that already exist ‚Äî `RunTurn` on an isolated `Session`, the `Emitter`,
> the `Approver` ‚Äî so the agent loop is **not touched**.
>
> Scope discipline: this is the **read-only, Explore-class** first slice. A
> writable subagent (with approval propagation) and parallelism are explicit
> non-goals here (¬ß4), deferred on purpose.

## 1. Background

Every phase so far made CodeAgent a more capable *single* agent. The one thing it
still cannot do is **delegate** ‚Äî push a side quest into a worker that does the
verbose, messy investigation and hands back only the conclusion. That is not
Claude-Code parity for its own sake; it is the **root fix for context hygiene**:

> Today, a "go figure out why this test fails" detour floods the *main* session
> with a dozen `read_file`/`grep` results the user will never look at again. They
> sit in history, get re-sent on every subsequent turn, and push the session
> toward compaction. The investigation's **byproducts** outlive its **conclusion**.

A subagent fixes this at the root: the detour runs in its *own* session, and only
its final answer comes back. The parent's context stays clean. This is the same
"keep exploration out of the main conversation" principle P7 applied to the
*renderer* ‚Äî here applied to the *model's context window*.

The pieces are all already in place, which is why this is a first slice and not a
phase of its own:

| Need | Already exists |
|---|---|
| Run a nested agent turn | `Runner.RunTurn(ctx, *session.Session, input)` ([loop.go:98](../internal/agent/loop.go)) ‚Äî depends only on an isolated `Session` |
| An isolated conversation | `session.NewBuilder(root)....Build()` ([builder.go](../internal/session/builder.go)) |
| Nested event correlation | `Event.SessionID` / `Event.TurnID` ([event.go](../internal/agent/event.go)) |
| Cost accounting for the subagent | the subagent's model calls go through the same `ResilientProvider`, whose `Observer` already records to the `requests` table ‚Äî **free** |
| Read-only vs side-effecting | the `SideEffecting` / `SideEffectingInput` markers ([side_effect.go](../internal/tools/side_effect.go)) |

## 2. What Claude Code actually does (verified, not recalled)

This slice was designed *against* CC's current behavior, checked against the
official docs rather than memory. The facts that shaped the decisions below:

- **Isolated context; only the final message returns.** "Each subagent runs in its
  own fresh conversation. Intermediate tool calls and results stay inside the
  subagent; **only its final message returns to the parent**." The parent gets it
  verbatim as the tool result (then may summarize).
- **Tools are an allow-list.** `tools` = "Array of **allowed** tool names. If
  omitted, inherits all tools." The read-only example is literally
  `tools: ["Read","Grep","Glob"]` ‚Äî "no Edit, Write, or Bash." An unpermitted call
  is **denied**, never silently run.
- **Each subagent has its own (required) system prompt** ‚Äî "tailored ‚Ä¶ with
  specific expertise, best practices, and **constraints**."
- **Each subagent can override the model** ‚Äî "Control costs by routing tasks to
  faster, cheaper models like Haiku." The built-in **Explore** agent is read-only
  and Haiku-by-default.
- **The prompt string is the only channel parent‚Üísubagent** ‚Äî no parent history,
  no parent system prompt. "Include any file paths, error messages, or decisions
  the subagent needs directly in that prompt."
- **Built-in Explore/Plan are one-shot** (not resumable). Our ephemeral v1 *is*
  this model.
- **Nesting has a depth cap** ‚Äî as of CC v2.1.172 subagents may spawn subagents,
  but one five levels below the main agent cannot spawn further; opt out by
  omitting `Agent` from `tools`. So **depth-1 is a choice, not a law.**

Sources: [Create custom subagents](https://code.claude.com/docs/en/sub-agents) ¬∑
[Subagents in the SDK](https://code.claude.com/docs/en/agent-sdk/subagents).

## 3. Goal

A `task` tool the model can call to delegate an exploratory subtask. The subagent
runs a nested `RunTurn` on an **isolated, ephemeral session** with a **read-only**
toolset and a **dedicated system prompt**, and returns **only its final
conclusion** as the tool result. The parent's context never sees the
investigation ‚Äî only the answer.

This first slice is deliberately the **Explore prototype**: read-only, cheap
model, one-shot. Read-only is not a limitation we regret ‚Äî it is the clean cut
that lets us ship without solving approval propagation (¬ß5.1).

## 4. Non-goals (v1)

- ‚ùå **Writable subagents.** Read-only only ‚Äî which means no approval propagation
  problem to solve yet. (The hardest part of the design, deferred on purpose.)
- ‚ùå **Parallel subagents** (`jobs`). One task runs synchronously to completion;
  the loop already executes tool calls sequentially.
- ‚ùå **Deep recursion.** The subagent's toolset excludes `task`, capping depth at 1.
- ‚ùå **Persisted / resumable sub-sessions.** Ephemeral, thrown away after the call
  (matches CC's one-shot Explore/Plan).
- ‚ùå **Any change to the agent loop, observation, reflection, or `Approver`/`Emitter`
  signatures.** If 8.3 ends with a diff inside the loop body of
  [loop.go](../internal/agent/loop.go), we did it wrong.

## 5. The design decisions (the crux)

Six decisions, each cross-checked against CC (¬ß2). These are where a naive "just
run a nested turn" goes wrong.

### 5.1 Toolset ‚Äî an explicit allow-list, default-deny (NOT a deny-list)

The unattended subagent runs with **no human to approve anything**, so its toolset
is a safety boundary, not a convenience.

> **Wrong (rejected):** derive the read-only set by *excluding* tools that
> implement the `SideEffecting` marker. Failure mode: a side-effecting tool that
> *forgot* the marker is treated as read-only, leaks into the subagent, and ‚Äî with
> no approver ‚Äî **runs unattended**. This is the opposite of our fail-safe-deny
> stance. (It is also a latent risk in the *main* loop already: an unmarked tool
> skips the gate there too.)

**Right:** the subagent's registry is an **explicit allow-list by name** ‚Äî
`read_file`, `list_files`, `grep`, `project_graph`, `git_diff`, `load_skill` ‚Äî
built via a new `tools.Subset(registry, names‚Ä¶)`. A *new* tool (read or write)
added later is **excluded by default** until someone deliberately adds it. That is
fail-closed by construction, and it matches CC's `tools: [Read, Grep, Glob]`.

**Defense in depth:** the sub-runner still gets a **fail-closed `Approver`** that
denies every side-effecting call ‚Äî *not* `nil`. The allow-list should make this
unreachable; the approver is the second wall if it ever isn't. (CC likewise
*denies* an unpermitted tool rather than running it.)

This also makes the **recursion guard explicit**: `task` is not in the read-only
allow-list, so a subagent cannot spawn a subagent. No depth counter needed for v1.

### 5.2 A dedicated, hard system prompt ‚Äî this is where subagents earn their keep

The subagent's final message lands in the **parent's** context verbatim. A verbose
answer defeats the entire purpose. CC's whole design leans on the subagent's
*system prompt* to enforce concision and role.

> **Wrong (rejected):** reuse the standard `prompt.AgentSystemPrompt` and bolt on a
> one-line "investigate and return a conclusion" wrapper. Too weak ‚Äî it neither
> compresses a rambling model nor stops it from trying to *ask the user* (a
> subagent has no user).

**Right:** a new `prompt.SubAgentSystemPrompt`, short and strict:
- return **only** the actionable conclusion, with `file:line` evidence;
- no preamble, no pleasantries;
- **never ask the user** ‚Äî there is none; decide or report what's missing;
- your answer is consumed by the parent's limited context ‚Äî be terse.

Seam: the session builder hardcodes the system prompt today
([builder.go:70](../internal/session/builder.go)). Add `Builder.WithSystemPrompt`
so the subagent session can use its own.

### 5.3 A cheaper model

Read-only `grep`/`read` investigation is the canonical "fast and cheap" workload ‚Äî
CC runs Explore on Haiku for exactly this. Reusing the parent's (possibly `-pro`)
model to walk files is waste.

**Right:** a config knob `agent.subagent_model` naming a configured model; the
sub-runner builds its provider + model from it. Unset ‚áí inherit the parent model
(safe default).

### 5.4 Return semantics when the subagent doesn't converge

The subagent has a small step budget (¬ß5.5). When it hits the cap,
`RunTurn` falls back to `finalAnswerAfterLimit`, whose worst case returns the
sentinel `"Agent stopped: reached the step limit‚Ä¶"`
([loop.go](../internal/agent/loop.go)). Returning *that* to the parent as a
"conclusion" is actively misleading.

**Right:** surface the limit explicitly ‚Äî expose `TurnResult.HitStepLimit` (or
detect the sentinel) and have the `task` tool return a structured
`"did not converge in N steps; partial findings: ‚Ä¶"` so the parent model can
decide to narrow the task and retry, rather than swallowing a non-answer.

### 5.5 Step budget

A fixed, small cap (`subAgentMaxSteps`, ~12) bounds the cost and latency of one
delegation. This is the analogue of CC's `maxTurns`.

### 5.6 Visibility ‚Äî default quiet

The whole feature exists to keep the main view clean ‚Äî for the **model's** context
*and* for the **human's** screen.

> **Wrong:** pipe the subagent's every `read_file`/`grep` event straight into the
> parent timeline. That re-pollutes exactly what we set out to keep clean.

**Right:** **default quiet.** The parent timeline shows only `task` start/finish
plus the final conclusion. The subagent's own tool events carry its distinct
`SessionID` (so they're already separable) and go to a collapsible sub-stream
surfaced behind a flag. Bracket the delegation with new `EventTaskStarted` /
`EventTaskFinished` kinds. (CC's default is likewise *not* to expand subagent
output inline.)

## 6. Architecture

```
internal/tools/task/            (new package)
   task.go      the `task` tool + the narrow SubAgent interface it depends on
internal/tools/registry.go      + tools.Subset(r, names‚Ä¶)  (allow-list helper)
internal/session/builder.go     + Builder.WithSystemPrompt(string)
internal/prompt/                + SubAgentSystemPrompt
internal/agent/event.go         + EventTaskStarted / EventTaskFinished
cmd/codeagent/subagent.go       (new) concrete SubAgent: builds an isolated
                                Session + read-only sub-Runner per call, runs one
                                turn, returns Final (or the ¬ß5.4 non-convergence form)
cmd/codeagent/main.go           buildRegistry: build the read-only subset, wire the
                                SubAgent, register the `task` tool into the PARENT
```

**Dependency direction (no cycle).** `internal/tools/task` imports only
`internal/tools` (for `Tool`/`ToolResult`) and defines its own narrow `SubAgent`
interface ‚Äî it does **not** import `agent`. The concrete `SubAgent` lives in
`cmd/codeagent`, which may import `agent` + `session` freely. `agent` imports
`tools` but never `task`. So the loop stays unaware a subagent exists.

```go
// internal/tools/task/task.go ‚Äî the tool only knows this interface.
type SubAgent interface {
    Run(ctx context.Context, prompt string) (string, error)
}

type Tool struct{ agent SubAgent }

func (t *Tool) Name() string          { return "task" }
func (t *Tool) SideEffects() bool      { return false } // read-only subagent
// InputSchema: { prompt: string (required) }  ‚Äî see ¬ß7 on the description.
func (t *Tool) Execute(ctx, input) (tools.ToolResult, error) {
    // parse prompt ‚Üí t.agent.Run(ctx, prompt) ‚Üí ToolResult{Content: conclusion}
}
```

```go
// cmd/codeagent/subagent.go ‚Äî a fresh isolated world per call.
func (s *subAgent) Run(ctx context.Context, prompt string) (string, error) {
    sess, _ := session.NewBuilder(s.root).
        WithBudget(s.subWindow, s.subThreshold).
        WithSystemPrompt(promptpkg.SubAgentSystemPrompt).  // ¬ß5.2
        Build()                                            // ephemeral, never stored
    sub := &agent.Runner{
        Model:    s.subProvider, ModelName: s.subModel,    // ¬ß5.3
        Tools:    s.readOnly,                              // ¬ß5.1 allow-list
        MaxSteps: subAgentMaxSteps,                        // ¬ß5.5
        Approver: denyAll{},                               // ¬ß5.1 fail-closed, not nil
        Observer: observation.DefaultObserver{},
        Emitter:  s.emitter,                               // ¬ß5.6 distinct SessionID
        // RemindSkills off: keep the subagent focused.
    }
    res, err := sub.RunTurn(ctx, sess, prompt)
    if err != nil { return "", err }
    if res.HitStepLimit {                                  // ¬ß5.4
        return notConverged(res), nil
    }
    return res.Final, nil
}
```

The parent's `buildRegistry` builds the read-only subset **before** it registers
`task` or the MCP tools, so the subset can never contain them (all MCP tools are
side-effecting anyway). The `task` tool is then registered into the **parent**
registry only.

## 7. The `task` tool's description matters

Because **the prompt is the only channel** to the subagent (¬ß2), the tool's
`Description()` must instruct the model to write a *self-contained* prompt:
inline the file paths, the error text, and the precise question, since the
subagent sees none of the parent conversation. This is the "soft" part ‚Äî like
skills, the feature's value depends on the model actually using it well, which the
description shapes.

## 8. Testing strategy

The seam makes this testable with **no live model and no terminal**:

- **`task` tool** ‚Äî fake `SubAgent`: assert the prompt is forwarded, the conclusion
  is returned as `ToolResult.Content`, and a `Run` error becomes a tool error.
- **`tools.Subset`** ‚Äî includes the named read-only tools; **excludes** anything
  side-effecting (`create_file`, `edit_file`, `git_commit`, every MCP tool) and
  `run_command` (which implements both markers,
  [run_command.go:250/260](../internal/tools/shell/run_command.go)); excludes
  `task` itself.
- **`Builder.WithSystemPrompt`** ‚Äî the built session's system message is the custom
  prompt, not `AgentSystemPrompt`.
- **subAgent integration** ‚Äî a fake `model.Provider` that returns text with no tool
  calls runs one turn; assert `Final` is returned and **no** session is written to
  the store. A second fake that always calls tools exercises the ¬ß5.4
  step-limit / non-convergence path.
- **fail-closed approver** ‚Äî `denyAll.Approve` returns false; a (hypothetical)
  side-effecting tool in the subagent is refused.

## 9. Phasing ‚Äî two PRs

- **PR1 ‚Äî mechanism, no live model.** `internal/tools/task` (tool + `SubAgent`
  interface) ¬∑ `tools.Subset` ¬∑ `Builder.WithSystemPrompt` ¬∑
  `prompt.SubAgentSystemPrompt` ¬∑ the `EventTaskStarted/Finished` kinds ¬∑ all unit
  tests above. Independently reviewable; changes no existing run path.
- **PR2 ‚Äî wiring.** `cmd/codeagent/subagent.go` (concrete `SubAgent`, read-only
  subset, fail-closed approver) ¬∑ `agent.subagent_model` config + a separate
  provider ¬∑ `buildRegistry` registers `task` ¬∑ default-quiet emitter handling ¬∑
  one real smoke test (delegate a read-only investigation, confirm only the
  conclusion returns).

## 10. Open decisions (recommendations; flag any to revisit)

1. **Read-only allow-list membership** ‚Äî `read_file, list_files, grep,
   project_graph, git_diff, load_skill` (recommended). `load_skill` is read-only
   (loads guidance); include it so a subagent can follow project skills.
2. **`subagent_model` default** ‚Äî inherit the parent model when unset
   (recommended), so the feature works out of the box; point it at a flash/haiku
   class model to get the cost win.
3. **Step budget** ‚Äî fixed `subAgentMaxSteps = 12` (recommended); tune from real
   runs.
4. **Visibility** ‚Äî default quiet, sub-stream behind a flag (recommended, ¬ß5.6).
   (The exact "expand subagent output" keybinding in CC was not verified; only the
   default-collapsed *direction* was.)
5. **Recursion** ‚Äî depth-1 via the allow-list excluding `task` (recommended for
   v1). CC allows depth-5; lifting our cap later is a deliberate, separate choice,
   not a prerequisite.
6. **Sub-session persistence** ‚Äî ephemeral, not stored (recommended; matches
   Explore/Plan). Persisting subagent transcripts separately (so they survive
   parent compaction, √Ý la CC) is a debuggability follow-on.

## 11. Why this is worth doing now

8.2 opened CodeAgent *outward* (external tools). 8.3 opens it *downward*
(delegation) ‚Äî and together they are the two items that move it from "a capable
agent" to "a platform." Of the two, the subagent is the higher architectural fit:
it is the root fix for the context-hygiene problem every long task has, and it
reuses `RunTurn`, the isolated `Session`, the `Emitter`, and the cost-recording
path wholesale. The read-only first slice ships that value while sidestepping the
one genuinely hard part (approval propagation) ‚Äî and, as with P7, the risk is
concentrated in one small, well-understood seam while the agent loop stays exactly
as it is.

**Done when:** the model can call `task("investigate why X fails")`, the subagent
finishes its read-only investigation in an isolated session on a cheap model, and
**only its conclusion** returns to the parent ‚Äî the dozens of investigation steps
never enter the parent's context, and the agent loop has not changed by one line.

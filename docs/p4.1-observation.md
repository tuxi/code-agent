# P4.1 — Observation Layer (design PRD)

> Status: design. No implementation yet. This document is the spec to hand to a
> developer. It deliberately defines a **data layer**, not a control machine —
> see "Guardrail" below.

## 1. Background

`run_command` already returns a structured result:

```json
{ "command": "go test ./...", "stdout": "...", "stderr": "...",
  "exit_code": 1, "duration_ms": 1840, "decision": "allow" }
```

That is the *raw material* for reasoning, but the model still receives it raw:

- An `exit_code: 1` with 800 lines of `go test` output buries the three lines
  that actually matter (`--- FAIL: TestParse`, `foo.go:42: undefined: Bar`).
- The loop truncates long output (~9.8k chars), so the real error can be cut off
  before the model ever sees it.
- Nothing tells the model *what kind* of failure this is (compile vs. test vs.
  lint vs. timeout), which is the first thing a human decides.

P4 ("the model autonomously runs tests, reads failures, and fixes them") cannot
work well until failures are **legible**. P4.1 makes them legible.

## 2. Goal

Add a thin layer that turns a raw `ToolResult` into a machine-actionable
`Observation`:

```
run_command → ToolResult {stdout, stderr, exit_code}
        ↓  observation.Observe()        ← pure, deterministic, no control
Observation {
   ok:           false,
   failure_type: "compile",
   summary:      "build failed: 1 error in internal/foo",
   salient:      ["internal/foo.go:42: undefined: Bar"]
}
        ↓  model reads it, decides the next tool call   ← control stays here
```

The model still owns every decision. The Observation only makes the decision
*well-informed*.

## 3. Guardrail (the most important section)

This project's spine:

> The model owns control flow. The runtime owns boundaries.
> The loop contains **no decision state machine**.

P4.1 names like "Retry Planning" / "Verify-Fix Loop" could be read as control
flow. They must not be built that way. Two shapes:

| | **Shape A — data layer (this PRD)** | Shape B — control machine (rejected) |
|---|---|---|
| What it produces | An `Observation` value handed to the model | A runtime that decides retries |
| Who retries | The model, via its next tool call | A state machine in the runtime |
| Verify-Fix loop | **Emergent** from the existing uniform loop | Hardcoded sequence |
| Matches philosophy | ✅ | ❌ the exact thing the README warns against |

**Decision: Shape A.** "Retry Planning" is reframed as *surfacing the failure and
an optional suggested next step as data*; the model pulls the trigger. The
Verify-Fix loop of P4.2 is then not new control code — it *emerges* because the
model reads the Observation and acts (`run_command` → read Observation →
`edit_file` → `run_command` again).

## 4. Non-goals

- ❌ A runtime retry/verify state machine or hardcoded fix sequencing.
- ❌ A separate "reflection engine" (that is P4.3, and stays grounded in real
  tool results).
- ❌ Coupling the thin agent loop to tool-specific semantics — enrichment lives
  behind an interface, injected like `Emitter` / `Approver` / `Compactor`.
- ❌ Re-running or mutating anything. Observation is read-only and pure.

## 5. Data model

```go
package observation

// FailureType is a coarse, model-friendly classification of why a tool result
// is not "ok". It is intentionally small; the model reads salient lines for
// detail.
type FailureType string

const (
    FailureNone    FailureType = "none"    // exit 0 / no failure
    FailureCompile FailureType = "compile" // build / type / syntax error
    FailureTest    FailureType = "test"    // test assertions failed / panic
    FailureLint    FailureType = "lint"    // vet / linter findings
    FailureRuntime FailureType = "runtime" // nonzero exit, not the above
    FailureTimeout FailureType = "timeout" // killed by the deadline
    FailureBlocked FailureType = "blocked" // refused by CommandPolicy
)

// Observation is the enriched, machine-actionable view of a single tool result.
type Observation struct {
    Tool        string      `json:"tool"`
    OK          bool        `json:"ok"`
    FailureType FailureType `json:"failure_type"`
    Summary     string      `json:"summary"`           // one line, model- and human-readable
    Salient     []string    `json:"salient,omitempty"` // the lines that matter, capped
}
```

**Resolved (sign-off):** P4.1 answers only *"what happened"*, never *"what to
do next"*. A `Suggestion` field was considered and **cut** — it carries an
"I think you should…" tone that slides the data layer toward a control layer,
which is the single most dangerous drift in this design. *What to do next* is the
model's job (P4.2), grounded in this Observation.

**Deferred (future, non-breaking):** a `Confidence float64` field. Classification
is heuristic — a bare `exit 1` is genuinely ambiguous between runtime and
compile — so telling the model *"92% sure this is compile"* is more honest than a
falsely absolute label. Nice-to-have; add once we have real failure transcripts
to calibrate against. Adding a field later does not break callers.

## 6. The Observer interface and where it plugs in

Mirror the existing injected-collaborator pattern (`Emitter`, `Approver`,
`Compactor`). The loop stays tool-agnostic: it calls one injected hook; `nil`
means "no enrichment", so behavior is unchanged when unset.

```go
package observation

// Observer turns a raw tool observation into a structured Observation. It is
// pure: same input → same output, no side effects, no control flow.
type Observer interface {
    Observe(tool string, rawObservation string) Observation
}
```

Integration point in `agent/loop.go` (today the loop builds an `observation`
string, truncates it, appends it to the session, and emits `EventToolFinished`).
The change is ~3 lines and contains no tool-specific logic:

```go
observation = TruncateObservation(observation, 9800)

// NEW: enrich (no-op when r.Observer == nil). Pure data; control stays below.
if r.Observer != nil {
    obs := r.Observer.Observe(call.Function.Name, observation)
    observation = obs.Render(observation) // prepend a compact summary block
    r.emit(Event{Kind: EventObserved, ToolName: call.Function.Name, Observation: obs.Summary})
}
```

`Render` prepends a short, fixed-format block so the signal sits *before* the
(possibly truncated) raw output the model already gets:

```
[observation] failure=compile  summary="build failed: 1 error in internal/foo"
  internal/foo.go:42: undefined: Bar
---
<original tool result follows>
```

Why prepend, not replace: the raw result keeps full fidelity for the rare case
the model needs it; the summary guarantees the salient lines survive truncation.

## 7. Failure classification (run_command first)

`Observe` parses `run_command`'s JSON; for other tools it returns a trivial
`OK`/not-OK Observation. Classification is heuristic and ordered:

1. `decision == "block"` → `FailureBlocked` (note carries the policy reason).
2. `note` mentions a timeout → `FailureTimeout`.
3. `exit_code == 0` → `FailureNone`, `OK = true`.
4. Otherwise inspect command + stderr/stdout:
   - compile markers — `undefined:`, `cannot use`, `syntax error`,
     `expected ...`, `not enough arguments`, `# <pkg>` headers → `FailureCompile`.
   - test markers — `--- FAIL`, `FAIL\t`, `panic:`, `--- FAIL:` → `FailureTest`.
   - lint markers — `go vet`, `golangci-lint`, `vet:` lines → `FailureLint`.
   - else → `FailureRuntime`.

The classifier is a pure function over `(command, stdout, stderr, exit_code,
note)`. It is per-tool-family extensible (Go first; `cargo`, `swift`, `pytest`
markers added later) but ships with the Go toolchain markers only.

## 8. Salient-line extraction (summarization)

Distill long output to the lines that carry signal, capped (e.g. ≤ 12 lines):

- Keep lines matching error markers (`error:`, `--- FAIL`, `panic:`,
  `undefined:`, `expected`, `cannot`, a `path:line:col:` diagnostic prefix).
- Preserve order; de-duplicate adjacent repeats.
- If nothing matches but exit is nonzero, fall back to the last N non-empty
  stderr lines (the shell convention that the error is at the end).

**Resolved (sign-off):** caps are `MaxSalientLines = 20`, `MaxLineLength = 300`
(up from the initial 12 / 200). Go compile failures routinely emit a dozen-plus
`file:line:col:` diagnostics; the goal is *enough signal in one shot*, not
maximal compression. Tune down later from telemetry if needed.

`Summary` is then a one-liner derived from `failure_type` + counts, e.g.
`"go test: 3 failed (TestParse, TestLex, …) in 2 packages"` or
`"build failed: 1 error in internal/foo"`.

## 9. Testing strategy

Entirely offline, table-driven over canned tool outputs — no model, no real
compiler:

- `classify_test.go`: `(command, stderr, exit_code) → FailureType` cases for
  compile / test / lint / runtime / timeout / blocked / none.
- `salient_test.go`: long `go build` / `go test` dumps → expected salient lines.
- `observe_test.go`: full `run_command` JSON → expected `Observation`.
- A loop test with a fake `Observer` asserting the rendered block is prepended
  and `EventObserved` is emitted — and that with `Observer == nil` the
  observation is byte-for-byte unchanged (proves the loop stays neutral).

## 10. How P4.2 (Verify-Fix) emerges for free

With P4.1 in place, no new control code is needed for the verify-fix loop:

```
model: run_command "go test ./..."
   → Observation{failure_type: test, salient: ["foo_test.go:10: want 3 got 4"]}
model: read_file foo.go → edit_file foo.go
model: run_command "go test ./..."
   → Observation{ok: true}
model: done
```

The loop already executes whatever the model asks, in whatever order. P4.1 just
makes each result legible enough that this sequence *emerges*. This is exactly
the existing Phase 4 "Done when: the model … fixes them **without any hardcoded
sequencing**."

## 11. Phasing

- **P4.1.a** — ✅ **shipped.** `Observation`, `FailureType`, the pure classifier,
  the salient extractor, `Observe`, `Render`. `internal/observation`, no loop
  change, fully table-tested offline.
- **P4.1.b** — ✅ **shipped.** `agent.Observer` interface (defined in `agent`
  alongside `Approver` / `Emitter`, returns `observation.Observation`),
  `Runner.Observer` injection, the loop wiring (Observe on full output → truncate
  → prepend), and `EventObserved`. Wired into both REPL and one-shot runners via
  `observation.DefaultObserver{}`. Loop tests cover enrichment, the emitted
  event, and the nil-Observer-is-a-no-op invariant.
- **P4.1.c** — ⏳ **partial.** The console relabels the tool-result line to
  `[result]` so the prepended `[observation]` block reads cleanly; a dedicated
  concise `EventObserved` console line is still open.

**Now: run it for real.** The point of finishing P4.1.b is telemetry. Drive a
week of real `go build` / `go test` / `go vet` cycles, then revisit §13 — the
classifier and caps should be tuned from transcripts, not from imagination.

## 12. Decisions (signed off)

1. **Render placement** — ✅ **prepend a summary block** (not a separate tool
   message). This is Context Engineering, not control flow: the signal sits
   first so it survives truncation and the model sees it at a glance.
2. **Suggestion field** — ✅ **cut from P4.1.** Observation describes *what
   happened*; *what to do next* is the model's call (P4.2). Keeps the data layer
   from sliding into a control layer. (See §5.)
3. **Cap sizes** — ✅ `MaxSalientLines = 20`, `MaxLineLength = 300`. (See §8.)
4. **Extensibility** — ✅ **single classifier**, no registry. `run_command` is
   the only tool that needs Observation today and ~95% of it is Go. Extract a
   per-tool registry only when gopls / SourceKit / rust-analyzer actually land.
   Designing for imagined future tools now is the wrong trade.

**Deferred:** `Confidence float64` on `Observation` (§5) — honest about
heuristic ambiguity; add once real transcripts exist to calibrate it.

## 13. Future work (recorded, not built)

Deliberately parked until a week of real-run telemetry says they are worth it —
building them now would be designing for imagined failures.

- **Shared marker taxonomy.** Today the classifier (`compileMarkers` /
  `testMarkers`) and the salient extractor (`salientSubstrings`) keep their own
  marker lists. As cargo / pytest / SourceKit land — and as more Go failure
  shapes show up (`PASS`/`SKIP`, `build constraints exclude all Go files`,
  `nil pointer dereference`, `duplicate symbol`, `segmentation violation`) —
  fold these into shared, categorized tables (`compile` / `test` / `runtime` /
  `lint`) used by *both*, so extension is one edit, not two.
- **Diagnostic window (extractSalient → extractDiagnosticWindow).** A line like
  `service.go:42:13: undefined: Bar` tells the model *where* but not *what
  surrounds it*. A future extractor can attach a few neighbor source lines
  (read from the file at `path:line`) so the model sees the enclosing function /
  struct without a separate `read_file`. This is a strict upgrade to salient
  extraction; the interface (`[]string` of context) is unchanged. Not P4.1.

- **Robust result detection (done).** `parseCommandResult` no longer substring-
  matches `"exit_code"`; it requires ≥ 2 of the signature fields
  (`exit_code` / `command` / `decision`) so another tool's JSON that merely
  mentions one of them is not misread as a command result.

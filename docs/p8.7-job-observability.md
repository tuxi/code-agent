# Phase 8.7 — Job & Subagent Live Observability — design PRD

> Status: **Phase A + B implemented; §8.4-2 bracket forwarding implemented**
> (job event partition + `job_wait` + brackets into the parent stream, live +
> replay). Phase C (live WS attach on the child stream) not started — §8.4

## 1. Background — the two reported problems

A user ran `npx skills add okx/onchainos-skills --yes -g` (a slow git clone) via
`run_command(background: true)`. Two things went wrong:

1. **The turn gave up mid-install.** The agent polled `job_status`/`job_logs` in a
   loop; each poll consumes one step of `Runner.MaxSteps` ([internal/agent/loop.go:178](../internal/agent/loop.go),
   `defaultMaxSteps = 24`). A slow clone burns the whole budget on polling alone —
   nothing to do with wall-clock time, since a background `jobs.Job` itself has no
   timeout ([internal/jobs/jobs.go](../internal/jobs/jobs.go)). At the limit, the loop
   forces one tool-free "answer from what you have" call ([loop.go:625-635](../internal/agent/loop.go))
   which is what produced the "check the terminal yourself" reply.
2. **The UI repeated `job_status job_1` / "still running" dozens of times.** This is
   not a rendering bug — the model genuinely re-invoked the tool every step because
   polling is the *only* way it has to check progress, and [internal/prompt/system.go:52-64](../internal/prompt/system.go)
   only *advises* sparse polling, it doesn't enforce anything.

Root cause for both: `internal/jobs` has **zero integration with the event
system**. A job's state changes are a mutable `Snapshot()`, pulled on demand, never
pushed. Compare with Claude Code's own subagent UX: a spawned subagent streams its
full step-by-step transcript to a side panel in real time, *without* the parent
agent polling anything and without that transcript ever entering the parent's
context or budget — the parent gets exactly one push notification when it's done.

## 2. What already exists (verified — not a rebuild)

Investigation turned up that this codebase already built the harder half of this
for a *different* feature (`task` subagent delegation, P8.3), and the primitives
generalize cleanly:

| Need | Already exists |
|---|---|
| An event log partitioned by an arbitrary id, no schema change required | `session_events` has no `FOREIGN KEY` to `sessions` ([internal/session/sqlite/store.go:96-104](../internal/session/sqlite/store.go)) — the subagent already writes events under a `session_id` (`sess.ID`) that has **no row in `sessions`**, proven by `internal/runtime/subagent.go:88-116` |
| A reusable event sink that persists under an arbitrary `SessionID` | `EventStoreEmitter{Ctx, Store, Next}` ([internal/runtime/observer.go:56-80](../internal/runtime/observer.go)) — `Emit` keys purely off `ev.SessionID`, which the caller stamps on the `agent.Event` it constructs |
| Fan-out to multiple sinks without touching the loop | `MultiEmitter` ([internal/runtime/emitter.go:9-14](../internal/runtime/emitter.go)) |
| Bracketing events for an async child unit | `EventTaskStarted` / `EventTaskFinished` ([internal/agent/event.go:49-50](../internal/agent/event.go)), pattern in `subagent.go:116,148` |
| Incremental replay of a partitioned stream by id | `GET /v1/conversations/{id}/events?since=N` → `loadEventsSince` → `eventStore.ReplaySince(ctx, id, sinceSeq)` ([internal/server/mux.go:259,414-420](../internal/server/mux.go)) — resolves purely by `id`, **does not require a live conversation or a `sessions` row** |
| Live chunk-level event emission (precedent, not yet used by jobs) | `EventToolStdout` / `EventToolStderr`, emitted per chunk keyed by `CallID` for **foreground** command output ([internal/agent/loop.go:832-835](../internal/agent/loop.go)) |

**What this buys us: Phase A below needs zero schema changes and zero new server
endpoints.** It's wiring, not new infrastructure.

## 3. What's genuinely missing (verified — do not assume solved)

Two things researched and confirmed **absent**, called out explicitly so this
doc doesn't overclaim:

1. **No push path from a finished async unit into the *parent's own* live
   stream.** `task_started`/`task_finished` are written only to the subagent's
   own sinks (`EventStoreEmitter` + a CLI-only `Progress` heartbeat) — never to
   the parent conversation's stream (`subagent.go:98-105`, explicit comment: *"NEITHER
   is the parent's live renderer"*). The parent only learns the subagent is done
   because `task` is an ordinary **synchronous** tool call — `tool_finished` on
   the `task` call itself is the notification. Background jobs are async, so
   there's no equivalent today: nothing tells the parent's timeline "job_1
   finished" unless the model calls `job_status` again. This is new plumbing, not
   reuse.
2. **No live WS attach to a non-root session.** `wsResolve` in `mux.go:282`
   resolves conversations through `conversation.Manager`, which only knows about
   root conversations created via `POST /v1/conversations`. A job or a subagent's
   sub-session is not registered there — `GET .../events` (poll/replay) works for
   any id because it reads straight from SQLite, but a real-time `/stream` socket
   does not exist for anything but a root conversation. **This is the piece that
   actually reproduces a live-updating sidebar; without it, a client can only
   poll the REST endpoint, which is a strictly better-than-today but not
   Claude-Code-equivalent experience.**

## 4. Proposed design

Three independently shippable phases, ordered by value/effort. **A and B directly
fix the reported bug; C is the architectural piece that matches the Claude Code
sidebar experience.**

### Phase A — Job event integration (durable, replayable; unblocks C for free)

Give `internal/jobs` the same treatment `subagent.go` already validated.

- **New `EventKind`s** in [internal/agent/event.go](../internal/agent/event.go):
  `job_started`, `job_output` (chunk, mirrors `tool_stdout`/`tool_stderr` — reuse
  the existing `Chunk` field), `job_finished`.
- **`jobs.Registry.Start` gains an optional sink.** Add a field (e.g.
  `Registry.OnEvent func(jobID string, ev jobs.Event)` or, more directly, thread
  an `agent.Emitter` into `Registry`) so `Start` can emit `job_started` and the
  completion goroutine ([jobs.go:130-147](../internal/jobs/jobs.go)) can emit
  `job_finished`. Keep `internal/jobs` free of an `internal/agent` import if we
  want to preserve its current zero-dependency package boundary — the cleanest
  shape is a small `type Sink interface { Started(jobID, command string); Output(jobID, chunk string); Finished(jobID string, status Status, exitCode int) }`
  defined in `internal/jobs`, with an adapter living in `internal/runtime` that
  turns those calls into `agent.Event{SessionID: jobID, ...}` fed through
  `EventStoreEmitter` — same shape as `RequestObserver` already does for
  telemetry ([internal/runtime/observer.go:13-41](../internal/runtime/observer.go)).
- **Live chunk emission**: replace `cmd.Stdout = out; cmd.Stderr = out` in
  `Registry.Start` ([jobs.go:121-122](../internal/jobs/jobs.go)) with a writer that
  both appends to the existing `cappedBuffer` (unchanged, still backs
  `job_logs`) and forwards each `Write` to the sink as a `job_output` chunk —
  same pattern already proven for foreground commands at `loop.go:832-835`.
- **Wiring**: `RegisterBuiltinTools` ([internal/runtime/registry.go:41](../internal/runtime/registry.go))
  currently builds `jobReg := runCmd.Jobs` with no access to `store`/`progress`
  (those only reach `BuildRegistry`, one level up, for `NewSubAgent`). Thread
  `store session.Store` into `RegisterBuiltinTools` (or return `jobReg` so
  `BuildRegistry` wires it after, mirroring how `NewSubAgent` is constructed
  *after* `RegisterBuiltinTools` returns today at `registry.go:141`) and attach
  the adapter there.
- **Net effect**: `GET /v1/conversations/{job_id}/events` becomes queryable
  **today, unchanged endpoint** — a job's full lifecycle (start, every output
  chunk, finish) is durable and replayable by id, exactly like a subagent's
  transcript is via `codeagent task-trace`. `job_status`/`job_logs` are
  untouched — they remain the on-demand "give me the current snapshot" tools,
  now redundant for *detecting* completion but still useful for *reading*
  output on request.

### Phase B — Fix the step-budget exhaustion directly

Phase A alone does **not** stop the model from re-polling — it only makes the
job's history durable. The budget fix is separate and smaller. Two options,
recommend the first:

- **Option 1 (recommended): a blocking `job_wait(job_id, timeout_seconds)`
  tool.** One call = one real wait via the already-existing `Job.done` channel
  (`select { case <-job.done: … case <-time.After(timeout): … }`,
  [jobs.go:54,78](../internal/jobs/jobs.go)) — costs exactly one step regardless
  of how long the wait is, vs. today's one-step-per-poll. Bounded timeout (e.g.
  default 30s, cap 90s so a single call can't itself stall a turn) so it always
  returns; the model can call it again if still running. This directly fixes
  "task abandoned after N steps of polling."
- **Option 2: exclude `job_status`/`job_logs` calls from `MaxSteps`.** Simpler
  patch (a name check in the loop's step counter) but weaker — an eager model
  can still spin in a tight, budget-free polling loop, which is arguably worse
  for latency/cost even if it no longer trips the step ceiling. Not
  recommended as the primary fix; could be a cheap belt-and-suspenders
  addition alongside Option 1.

### Phase C — Live WS attach for a child stream (the actual "sidebar" parity)

The bigger, architectural piece — applies equally to jobs **and** to `task`
subagents (neither has this today, per §3.2):

- Extend `wsResolve` ([mux.go:282](../internal/server/mux.go)) to resolve
  non-root ids: either (a) register a job/subagent sub-session with
  `conversation.Manager` under a distinct namespace so `Resolve` finds it, or
  (b) add a parallel lightweight path (e.g. `/v1/jobs/{id}/stream`) that wraps
  the job's live sink directly in a `StreamEmitter` without going through the
  full `Conversation` machinery (jobs have no message history, no turns — a
  full `Conversation` is more than they need).
- On `job_started`/`task_started`, forward a bracket event into the **parent's**
  own live stream carrying the child `id`, so the client discovers the child
  stream inline without needing to already know it exists. **Precision on what's
  wired today** (corrected during implementation — an earlier draft overclaimed
  the gap): the CLI heartbeat IS wired for run/repl via `subagentProgress()`
  ([cmd/codeagent/main.go:476,537](../cmd/codeagent/main.go), TTY only), and the
  TUI passes its `backend.Emitter` as `progress`
  ([cmd/codeagent/main.go:585](../cmd/codeagent/main.go)) — so run/repl/tui do
  receive `task_started`/`task_finished` live. What's missing is exactly the
  **serve path**: both `BuildRegistry` call sites that back WS clients
  ([internal/embed/server.go:320](../internal/embed/server.go),
  [cmd/codeagentd/main.go:89](../cmd/codeagentd/main.go)) pass `progress=nil`,
  so **no WS client (macOS/iOS) ever receives `task_started`/`task_finished`** —
  they only see the ordinary `tool_started`/`tool_finished` bracket for the
  `task` call itself. Closing that serve-path gap belongs in this Phase C pass —
  see §8.
- This is the piece that turns "poll a REST endpoint every second" (Phase A's
  ceiling) into genuine push-based real-time rendering, matching the
  screenshot. Recommend scoping as its own follow-on once A/B are landed and
  validated against the real bug report.

## 5. Open question — flagged, not solved by this doc

**Phase A/C give the *UI* a push-based view. They do not give the *model* one.**
This codebase's turns are synchronous request/response (`SendMessage` →
`RunTurn` → returns); there is no scheduler that can wake a session and start an
unsolicited turn when a background job completes mid-idle. Claude Code's own
harness *does* do this (a finished background agent injects a `task-notification`
that resumes the parent without it polling) — but that requires an async-wake
capability this runtime does not have. Phase B (`job_wait`) sidesteps the
question by keeping the wait inside a single synchronous tool call rather than
requiring out-of-band wakeup; closing this gap properly (autonomous resume on
job completion, no model-in-the-loop wait at all) is a larger scope item and
should not be assumed as a byproduct of A/B/C.

## 6. Non-goals (this phase)

- Parallel/concurrent job execution semantics beyond what `jobs.Registry`
  already allows (multiple jobs can already run concurrently; this phase is
  about observability, not scheduling).
- Lifting the subagent depth-1 cap or adding writable subagents — orthogonal,
  already tracked as P8.3 follow-ons.
- Autonomous mid-idle turn resumption (§5) — flagged as future work, not
  in scope here.
- Changing `job_status`/`job_logs` behavior or signatures.

## 7. Rollout order

1. **Phase A** (event integration) — no schema change, low risk, unblocks
   everything else. Ship first.
2. **Phase B, Option 1** (`job_wait`) — small, directly closes the reported bug.
   Can ship in parallel with A.
3. **Phase C** (live WS attach) — bigger lift, do once A/B are validated against
   a real repro of the original bug report (re-run the `onchainos-skills`
   install and confirm the turn no longer exhausts its step budget).

**Done when:** the exact repro from the bug report — a slow background install —
completes without the model exhausting `MaxSteps` on polling, and the job's full
lifecycle is queryable via `GET /v1/conversations/{job_id}/events` without any
`job_status` calls at all.

## 8. Host（AgentKit / macOS / iOS）侧契约（写给客户端团队对齐，不在本仓改）

### 8.0 现状基线（已核实，纠正一个此前的错误假设）

在写这一节之前核实了"客户端今天到底能看到什么"，发现比预期更少：

- **`run_command(background)` + `job_status`/`job_logs`**：客户端今天看到的就是普通
  `tool_started`/`tool_finished` 卡片（`tool_name` 分别是 `run_command`/`job_status`/
  `job_logs`），这正是 bug 报告里刷屏的那些卡片。这条路径**协议层面不用变**——Phase A/B
  上线后，这些工具调用**本身还会存在**（`job_status`/`job_logs` 不下线，只是不再是判断
  "完成没"的唯一手段）。
- **`task` 子agent（P8.3）内部**：**WS 客户端（macOS/iOS）目前完全看不到**，连
  `task_started`/`task_finished` 这两条 bracket 事件都看不到——尽管协议文档
  （`client_integration_v1.md:170-171`）和 golden 测试都已经定义好了这两个 `kind` 的线格式。
  原因：serve 路径的两个 `BuildRegistry` 调用点
  （[internal/embed/server.go:320](../internal/embed/server.go)、
  [cmd/codeagentd/main.go:89](../cmd/codeagentd/main.go)）把 `progress` 传的是 `nil`。
  （更正：CLI 的 run/repl 和 TUI **是**接了线的——run/repl 在 TTY 下走 `subagentProgress()`
  单行心跳，TUI 直接传 `backend.Emitter`——缺口只在 serve。）WS 客户端今天对 `task` 调用
  只能看到外层普通的 `tool_started`/`tool_finished`（`tool_name="task"`），中间过程是黑盒，
  结论在 `tool_finished.observation` 里一次性给出。

**结论：这次要补的不只是 job 的可观测性，还要顺手把 subagent 的观测线接上**——两者用
同一套客户端渲染逻辑，值得一起对齐。

### 8.1 Phase A 对客户端的影响 —— 协议增量，向后兼容，无强制改动

- 新增 `EventKind`：`job_started` / `job_output` / `job_finished`。按
  `client_integration_v1.md §7`："同一大版本内只增不改"——不升 `protocol_version`，旧客户端
  按 §5.5 规则忽略未知 `kind` 即可，**不改也不会崩**。
- 但 Phase A 单独上线时，这些事件**只写进 job 自己的分区**（`session_id` = job 的 id，
  和 `task_started` 的写法一致），**不会自动出现在父会话正在订阅的 WS 流里**——客户端要看到
  它们，必须已经知道这个 job id 并主动去 `GET /v1/conversations/{job_id}/events` 拉取。
  见 §8.4 决策点 2：建议同时把 bracket 事件转发进父会话的直播流，否则客户端无从发现 job 的
  存在。
- 建议客户端侧先做的事：在事件 reducer 里加上这三个新 `kind` 的 no-op-safe 处理（哪怕暂时
  不渲染），为 Phase C 的实时视图预留 reducer 分支，避免后续再动一次聚合逻辑。

### 8.2 Phase B 对客户端的影响 —— 基本无感

- 新增/暴露一个工具 `job_wait(job_id, timeout_seconds)`。对客户端来说就是又一张普通工具卡
  （`tool_started`/`tool_finished`，`tool_name="job_wait"`），复用现有的工具卡状态机
  （`client_integration_v1.md §3.3` "聚合身份"），**不需要新逻辑**。
- 可选体验优化：识别 `tool_name=="job_wait"` 时渲染成"正在等待任务完成…"而不是原始 JSON
  参数——纯 UI 糖，不阻塞其他 phase。

### 8.3 Phase C 对客户端的影响 —— 真正的新增能力，需要提前规划 UI

这是唯一需要客户端写**新组件**、而不只是扩展 reducer 的部分：

1. **发现子流**：从父会话收到的 `job_started`（以及补线后的 `task_started`）事件里取出子流
   id，决定要不要提供"展开查看详情"的入口（对应截图里的右侧栏触发点）。
2. **两种取数方式，建议分两步接**：
   - **立刻能做（依赖 Phase A，不依赖 Phase C 的后端改动）**：轮询
     `GET /v1/conversations/{child_id}/events?since=N`（**这个接口今天已经能用**，见
     [mux.go:414](../internal/server/mux.go)，按 id 直读事件日志，不要求是根会话）。可以先把
     侧边栏 UI、reducer、渲染逻辑都建好，用轮询驱动。
   - **Phase C 后端就绪后替换传输层**：换成真正的 WS attach（具体端点形态见 §8.4 决策点
     3），渲染逻辑不用变，只是从"轮询"换成"推送"。
3. **新 UI 组件**：一个可独立 attach/展开的"子流查看器"（侧边栏或行内展开），复用父会话已有
   的工具卡渲染器渲染子流内部的 `tool_started`/`tool_finished`/`job_output` 序列——这是对齐
   截图体验的核心组件，建议现在就排期，不必等 Phase C 后端。

### 8.4 需要客户端团队一起拍板的决策点（后端还没锁死线格式，等对齐后再冻结 golden）

1. **子流 id 放哪个字段**：复用 `session_id`（和 `task_started` 一致，但语义上 job 并不是
   一个 session，可能让客户端 DTO 命名别扭）vs. 新增一个专门的 `job_id` 字段（语义更清晰，
   但 `task_started`/`job_started` 就不共享同一个"子流 id"字段名了，reducer 要多判断一个
   字段）。**已定稿（2026-07-02）：复用 `session_id`**，golden 按此冻结（见 §8.5 Q2）。
2. **bracket 事件要不要转发进父会话直播流**：**已定稿"转发"并实现（2026-07-02，客户端确认
   密度可接受）**。实现形态：`job_started`/`job_finished` 双写——job 自己的分区一份（子流回放），
   父会话分区一份（行的 session_id=父会话，payload 的 `session_id` 仍是 job id，和
   `task_started` 携带子 id 的约定一致；`turn_id`=发起 turn，供入口卡落位）；serve 下同时推给
   父会话的直播订阅者，**直播帧盖的是父分区的 seq**（客户端的按会话游标不被破坏）。
   `job_output` 不进父流。见 `internal/runtime/jobs_sink.go` 的 `publish`。
3. **Phase C 的子流端点形态**：**已定稿"独立轻量端点"**（如 `/v1/jobs/{id}/stream`），附加两个
   客户端条件，后端实现时保证：(a) 事件帧格式与 `/v1/conversations/{id}/events` 完全一致（同一
   `WireFrame` 解码器直接用）；(b) job 不出现在 `GET /v1/conversations` 列表——**(b) 今天已由
   构造保证**：列表来自 sessions 表（`repo.List`），job 只写 `session_events`，永远没有
   sessions 行。

### 8.5 客户端追问的两个问题——已确认（2026-07-02）

**Q1：`since=N` 的 N 从哪来？**
`seq` 字段**已经在线格式上**（v1.2 §4 落地）——`client_integration_v1.md` 旧文
"§5.8 无 seq 级续传 / seq 是 v2 预留"是过时信息，已订正。规则：

- **N = 已收到帧里最大的 `seq`**，不是条数。`seq` 会话内单调递增但**有空洞**
  （底层是跨会话共享的 SQLite 自增 rowid，多会话交错写入必然跳号），用条数推算
  必错。
- `GET /events` 回放帧逐帧带 `seq`（[mux.go:291](../internal/server/mux.go) 从
  rowid 盖戳）；WS 直播帧同样带（[executor.go 的 sequencingEmitter](../internal/conversation/executor.go)，
  直播与回放同源同值）。`token_delta` 瞬态不带。
- 对 job 分区轮询同理：首次 `since=0`（或省略）全量，之后带上最大 `seq`。

**Q2：`job_finished` 的终态字段清单（golden 已冻结：
[testdata/job_finished.json](../internal/server/testdata/job_finished.json) 等三个）**

| 字段 | 出现条件 | 含义 |
|---|---|---|
| `kind` | 恒有 | `"job_finished"` |
| `at` | 恒有 | RFC3339 毫秒 UTC |
| `session_id` | 恒有 | **job id**（§8.4 决策点 1 按此定稿：复用 `session_id`） |
| `seq` | 回放/直播 | 同 Q1 |
| `text` | 恒有 | 终态：`exited`（成功）\| `failed` \| `canceled` |
| `elapsed_ms` | >0 时 | job 运行时长（毫秒） |
| `exit_code` | 仅失败 | **结构化退出码**（应客户端要求新增）：>0 = 命令非零退出；-1 = 启动失败或被信号杀死；成功时为 0 → omitempty 省略，`text=="exited"` 即代表成功 |
| `err` | 仅失败 | 人读描述（`"exit code 2"`），与 `exit_code` 冗余、供直接展示 |

入口卡/查看器顶栏的三分支显示建议：`exited` → 成功；`failed` → 失败（`exit_code`
细分：-1 显示"被终止/启动失败"，>0 显示"退出码 N"）；`canceled` → 已取消（主动
停止，样式上区别于失败）。`job_started.text` = 完整命令（入口卡标题可用）；
`job_output.chunk` = 合并后的输出片段（服务端已按 ~4KB/750ms 合并，客户端无需
再节流）。

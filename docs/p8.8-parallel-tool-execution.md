# Phase 8.8 — Parallel Tool Execution — design PRD

> Status: **core + orchestration guidance implemented, dark-launched**
> (2026-07-04). The read-parallel/write-serial batch executor is in
> `internal/agent/tool_batch.go`, gated by `agent.max_parallel_tools` which
> **defaults to 1 (exact old sequential behavior)** — flip the default after soak
> (§11). §7 orchestration guidance ships as a first-call reminder gated on the
> same knob. Frontend adaptations (§8) not yet done; they are only exercised when
> a deployment raises the knob. This is **Layer 1** of the multi-agent
> parallelism work: the general primitive from which "map" subagents fall out as
> a special case (a subagent is just the `task` tool). "loop" (dependent
> subtasks) already works — the main agent's own loop sequences them by threading
> each result into the next turn — so this phase does NOT build a workflow DSL;
> it makes the tool-call *batch* run concurrently. Companion to
> [p8.3-subagent.md](p8.3-subagent.md). Design values inherited from the README:
> *"Workflow is emergent, not encoded"* ([README.md](../README.md)) — the model
> expresses map (one batch) vs loop (across turns) through how it batches tool
> calls; we do not encode a graph.

## 1. Background & goal

Today the loop executes a tool-call batch **strictly sequentially**
([loop.go:501](../internal/agent/loop.go) — `for idx, call := range resp.ToolCalls`).
So when a model emits 5 `task` calls in one turn (the natural "fan out 5
subagents" gesture), they run one after another: correct result, but 5× wall
clock. Parallelism is the single change that unlocks the map pattern — and
because `task` is an ordinary tool, the same primitive also parallelizes plain
read-only work (read 5 files, grep + web_search at once).

**Goal:** execute independent tool calls in a batch concurrently, safely,
without changing the wire contract or the sequential semantics the model relies
on for dependent work.

**Explicit framing (from the design discussion):**
- **map** = independent subtasks = one batch of N calls run concurrently → this
  phase.
- **loop** = dependent subtasks (shot N needs shot N-1's last frame) = the main
  agent's existing loop, threading results across turns → already works, no code.
- Parallel *subagents* are not a subagent feature — they are a consequence of
  parallel *tool execution*. Build the primitive, get the feature.

## 2. What the industry does (verified against the pattern, not recalled)

- **OpenAI / Anthropic model APIs**: a single assistant message may carry
  multiple `tool_calls`; the harness is expected to run them concurrently and
  return one tool result per call. Parallel tool calling is a first-class model
  capability, not a framework bolt-on.
- **Claude Code**: parallelizes tool calls from one message; the safety rule that
  matters — **read-only calls run concurrently, mutating calls are serialized**
  (you never want two concurrent edits to the same file, or two `bash` with
  ordering intent racing).
- **Anthropic multi-agent research system**: a lead agent spawns subagents that
  run in parallel — and its own published caveat: multi-agent burns **~15× the
  tokens** of a single agent. Parallelism pays off only when subtasks are
  genuinely independent AND heavy.
- **LangGraph**: the opposite school — an explicit graph (Send API for map,
  edges for sequence). Deterministic but rigid. We deliberately do NOT follow
  this: it fights the emergent-workflow value.

**Consensus we adopt:** the model expresses the dependency structure by *how it
batches* tool calls; the runtime parallelizes what is safe to parallelize and
serializes what is not. No graph, no DSL.

## 3. Design principles

1. **Wire contract unchanged.** Events already key on `call_id` as a state
   machine (reducer guidance in [client_integration_v1.md](client_integration_v1.md)
   §3.3: "tool card is a state machine keyed on call_id, update-in-place, never
   assume tool_finished is adjacent to its tool_started"). Interleaved events
   from concurrent calls are already what that contract anticipated — the
   `call_id` work (commit 658c235 lineage) was the prerequisite, now paying off.
2. **Read-parallel, write-serial (the safety heart).** Never run two
   side-effecting calls concurrently, and never overlap a side-effecting call
   with anything. The marker is already there: `tools.HasSideEffectsFor(tool, input)`
   ([side_effect.go](../internal/tools/side_effect.go)) — input-aware, so
   `git status` parallelizes while `rm build` is a barrier.
3. **Preserve the model's ordering semantics.** Do not reorder the batch. The
   model's order encodes intent; we only parallelize *adjacent* safe calls.
4. **Goroutines are pure; the main goroutine owns shared state.** Concurrent
   workers compute a result into their own slot and touch no shared loop state
   (`turn.Steps`, `sess.Messages`, budgets); the main goroutine assembles them in
   order after the join. This sidesteps almost every data race by construction.
5. **Opt-out to exact current behavior.** A `max_parallel_tools = 1` config
   restores today's strictly-sequential loop, byte-for-byte.

## 4. Core mechanism

### 4.1 Batch partitioning — order-preserving barrier model

Walk `resp.ToolCalls` in order and cut it into **runs**:

- A maximal run of **consecutive read-only** calls → one **concurrent group**.
- Each **side-effecting** call → a **barrier**: it runs alone, after the prior
  group has fully joined, before the next group starts.

Example — model emits `[read A, read B, write C, read D, read E]`:
```
group {A, B}  ─join─►  barrier C  ─►  group {D, E}  ─join─►  done
   (concurrent)          (alone)        (concurrent)
```
For the map case — `[task, task, task, task, task]` (subagents are read-only) —
the whole batch is ONE group of 5: full parallelism.

**Consequence (important for frontends):** at most one side-effecting call is
ever in flight → **at most one approval pending at a time** → clients need no
approval-queue UI (§8).

### 4.2 Bounded concurrency

A semaphore caps in-flight workers at `max_parallel_tools` (default proposal: 8;
subagent-heavy batches also bounded by provider rate limits and token cost).
Exceeding the cap queues within the group. Prevents "5 subagents × their own
model calls" from stampeding the provider.

### 4.3 Result reassembly

Each worker writes `results[idx]` (observation / output / assets / err / elapsed).
After a group joins, the main goroutine appends the tool-result messages to
`sess.Messages` **in idx order** (the provider requires tool results to
correspond to the `tool_calls`; completion order must not leak into history) and
appends `turn.Steps` in order. `webSearches` budget is counted here (serial),
not in workers.

### 4.4 Interaction with existing loop guarantees

- **Cancel mid-batch (v1.2 §2.2):** on `ctx` cancel, stop launching new groups;
  let in-flight workers unwind (they observe ctx), then fill every not-yet-run
  call with `toolInterruptedObservation` in order, keeping history balanced and
  resumable — same invariant as today ([loop.go:500-512](../internal/agent/loop.go)),
  just applied after a concurrent join instead of a sequential break.
- **Mid-turn checkpoint (v1.2 §2):** unchanged boundary — checkpoint after the
  batch (all groups) joins. Still a consistent point.
- **Hooks (8.5):** pre/post hooks run inside each worker; they are independent
  `sh` processes, safe to run concurrently. Post-hooks that mutate (gofmt) only
  attach to side-effecting calls, which are serial — no concurrent writes.
- **Reflection / budgets:** unchanged (they act at batch boundaries / on the
  serial assembly path).

## 5. Concurrency safety audit (what must be verified)

| Shared resource | Status | Action |
|---|---|---|
| `sess.Messages`, `turn.Steps`, `webSearches`, `turnAssets` | mutated per call | **Workers never touch them**; main goroutine assembles post-join (§4.4). Root of the design. |
| `r.emit` → sequencingEmitter → store + bus | store is WAL+mutex; `sessionBus.Emit` locks; wsSink has a write mutex | Likely safe; **must add a `-race` test** for concurrent emit. seq interleaves — fine (monotonic, atomic at persist; clients key on call_id). |
| Approver | `RemoteApprover.pending` is a `map[id]` under mutex — already multi-pending | With write-serial (§4.1) at most one approval at a time, so this is moot, but the approver is structurally ready if we ever relax it. |
| token_delta streaming | only the parent's own model call streams; tools/subagents don't stream to the parent | No interleave — subagent model calls go through their own emitter, not the parent's preview. |
| Provider / rate limits | N concurrent subagent model calls | Bounded by the §4.2 semaphore + a cost surfacing (§9). |

## 6. Scheduling policy = the safety contract

The partitioning in §4.1 IS the safety policy, stated as an invariant:

> **A side-effecting tool call never executes concurrently with any other tool
> call.** Read-only calls execute concurrently up to the semaphore bound. The
> batch's observable effects are identical to sequential execution; only
> independent reads are accelerated.

This is provable from the barrier model and is the property the frontends and the
approval UX rely on.

## 7. Orchestration guidance — the harder, more important half — IMPLEMENTED

Mechanism alone regresses to "parallel chaos" (the article's warning; Anthropic's
15×). The system prompt must teach the model **when** to fan out. Shipped as a
**gated first-call reminder** (`parallelReminder` in
[loop.go](../internal/agent/loop.go), injected on the first model call of a turn
via `Runner.RemindParallel`), mirroring the `skillsReminder` pattern:

- Fan out (one batch of `task` calls, or several reads/searches) ONLY when
  subtasks are **independent** AND each is **heavy** — "research these 5
  unrelated topics", "check security / perf / tests / style at once".
- Do NOT fan out trivial or dependent work: "Parallelism is for breadth, not for
  depth."
- For a **dependent** chain (loop: shot N needs shot N-1), do one call, use its
  result, then the next — do not batch.

**Key decision: gated on `MaxParallelTools > 1`.** `BuildRunner` sets
`RemindParallel` only when parallelism is actually enabled. With the knob at the
default (1), fanning out would burn tokens (5 serial subagents) with zero
speedup, so the guidance is withheld — the model is never told to do something
the runtime won't accelerate. Ephemeral (re-injected each turn, never persisted),
so it costs nothing once the model isn't fanning out.

## 8. Frontend adaptations (TUI / CLI / macOS / iOS)

The wire is unchanged, but "multiple tools in flight at once" is a new runtime
reality each frontend must handle. Summary: **the WS clients are structurally
ready; the terminal frontends need real work.**

### 8.1 macOS / iOS (WS) — structurally ready, UI enhancements

- **Reducer: already correct.** Tool cards are a state machine keyed on
  `call_id`, update-in-place. N concurrent `tool_started` (no matching
  `tool_finished` yet) → N cards simultaneously in "running" — this already
  works. **No reducer change.**
- **Must confirm:** the client does NOT assume `tool_finished` immediately
  follows its `tool_started` (some naive renderers pair adjacent events). The
  contract already forbids this; verify the implementation honors it.
- **Approval:** with write-serial (§4.1), at most one `approval_request` is ever
  pending, so the existing single-approval UI stays valid. **No approval-queue
  work needed** — call this out to the client so they don't over-build.
- **Enhancements (optional):** visually indicate a concurrent batch (e.g. group
  the running cards, or a "×5 running" affordance); the cost/footer meter already
  aggregates across concurrent model calls, just burns faster.
- **childStream (P8.7) synergy:** 5 parallel subagents = 5 childStream entry
  cards appearing together, each openable — the P8.7 viewer already handles this
  per-`call_id`.

### 8.2 TUI — real work

- **Concurrent in-flight rendering:** today's status line/heartbeat assumes ONE
  active tool. It must become a list of N in-flight tools (or a compact "N
  running" with expandable detail).
- **Thread-safe event intake:** the TUI backend emitter now receives events from
  multiple goroutines. bubbletea is single-goroutine (message loop), so events
  must be marshaled onto its `Program.Send` channel — verify the current
  `backend.Emitter` already funnels through a channel (it should); add a `-race`
  test.
- **Subagent heartbeat rework:** `taskProgress`
  ([cmd/codeagent/subagent_progress.go](../cmd/codeagent/subagent_progress.go))
  is explicitly single-goroutine, single in-place line ("emits arrive
  sequentially"). With parallel subagents, N heartbeats would fight over one
  line. Rework to N lines (or a counter), or disable it when a batch is parallel.

### 8.3 CLI (`run` / one-shot) & REPL — minor

- Linear stdout: concurrent tools **interleave** their output. Tolerable, but the
  per-tool framing must not assume "start … end" are adjacent (prefix lines with
  a short call id, or buffer per-call and flush on finish).
- Same `taskProgress` single-line issue as the TUI (run/repl share it) — rework
  or disable under parallelism.
- `-race` the run/repl path since emits now cross goroutines.

### 8.4 What NONE of them need

- No wire/DTO change, no protocol version bump (call_id + seq already carry
  everything).
- No approval-queue UI (write-serial guarantees ≤1 pending).
- No new event kinds.

## 9. Config & knobs

- `agent.max_parallel_tools` (int, default proposal 8). `1` = today's exact
  sequential behavior (the kill switch / rollback).
- Consider `agent.max_parallel_subagents` as a tighter sub-bound (subagents are
  the expensive case: each is a full nested turn). Or fold into the single knob
  for v1.
- Surface concurrency in the cost report so the 15× risk is visible.

## 10. Non-goals (this phase)

- **Reordering** the model's batch or inferring a dependency DAG — order is the
  model's contract; we only parallelize adjacent read-only runs.
- **Parallel side-effecting** tools — deliberately serialized (safety + single
  approval). A future phase could relax with per-resource locking, not now.
- **Agent Team** (shared task board, peer messaging, roles) — a different
  architecture (shared mutable blackboard, not a parent-child tree). Explicitly
  out of scope; this phase stays within the orchestrator-worker (subagent)
  paradigm.
- **Cross-turn / background parallelism** — already covered by jobs (P8.7) for
  shell work; this phase is about the in-turn batch.

## 11. Rollout & testing

1. **Core (behind `max_parallel_tools`, default 1 first):** partitioning +
   bounded concurrent group execution + ordered reassembly. Land dark, flip
   default after soak.
2. **Safety tests:** a batch of mixed read/write asserts write-serial (no two
   side-effecting overlap; observable effects == sequential); `-race` on
   concurrent emit; cancel-mid-group keeps history balanced & resumable;
   ordered reassembly under shuffled completion.
3. **Map E2E:** 5 read-only `task` calls in one batch run concurrently, results
   fan back in order, parent aggregates — measure wall-clock vs sequential.
4. **Frontends:** TUI multi-in-flight + heartbeat; CLI interleave framing; a WS
   integration test asserting N concurrent tool_started render as N running
   cards (client-side, but back it with a server test emitting concurrently).
5. **Orchestration guidance:** prompt eval — does the model fan out on
   independent-heavy tasks and stay serial on dependent ones?

## 12. Open questions

- Default for `max_parallel_tools` — 8? Tie to provider concurrency limits?
- Should `task` get a convenience "batch" form (one call, array of prompts) for
  models weak at parallel tool-calling, or rely purely on parallel `tool_calls`?
  Lean: rely on parallel calls (emergent); add batch form only if evals show
  weak models under-parallelize.
- Do we bound subagents separately from plain tools (they are far heavier)?
- Cost governance: hard cap on concurrent subagent token spend per turn?

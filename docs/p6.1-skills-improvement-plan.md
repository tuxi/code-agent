# Phase 6.1 — Skills improvement plan

> Status: plan. Written 2026-06-30 after comparing code-agent's skill implementation
> against the Claude Code skills repository (`anthropic-agent-skills`). Companion to
> [docs/p6-skills.md](./p6-skills.md) which defines the original architecture.

## 0. Context: what we compared

| Surface | code-agent | Claude Code (`anthropic-agent-skills`) |
|---|---|---|
| Core format | Markdown + YAML frontmatter (`---`) | Same |
| Required fields | `description` | `name` + `description` |
| Optional fields | `version` | `license` |
| Directory layout | `skills/<name>/SKILL.md` **and** `skills/<name>.md` | `skills/<name>/SKILL.md` only |
| L1 (always in context) | `PromptIndex()` → system prompt text block | `available_skills` structured list |
| L2 (on trigger) | Model calls `load_skill(name)` tool → body returned | Body auto-injected into context |
| L3 (resources) | ❌ Not implemented | `references/`, `scripts/`, `assets/` — read/run on demand |
| Distribution | Manual file placement | Plugin marketplace + `.skill` packaging + API |
| Skill creation DX | None | `skill-creator` skill + eval framework + description optimizer |
| Eval / quality | Go unit tests only | Baseline comparison, A/B blind test, statistical benchmarks |

**Verdict:** The core format is compatible. The same `SKILL.md` works in both systems.
The gaps are in (1) L3 resources, (2) distribution, and (3) authoring toolchain.

---

## 1. Improvement tasks — by priority

### P0 — Compatibility & correctness (blockers for real-world use)

#### Task 0.1: Require `name` in frontmatter

**Why.** Claude Code skills always carry a `name`. code-agent currently falls back to
the directory/filename when `name` is absent. That produces different identities for
the same skill file depending on how it's stored — a `SKILL.md` with no `name` in a
directory called `foo` becomes `foo` in code-agent but would be rejected (or
unnamed) in Claude Code. This is a silent interop bug.

**What.**
- In `parseSkill()`, treat a missing `name` as a hard parse error (record in
  `Skipped`), not a silent fallback.
- Remove the `if skill.Name == "" { skill.Name = label }` fallback in `loadDir`.
- Update `TestLoad_NameFallback` → `TestLoad_MissingNameIsSkipped`.
- Update `TestLoad_BareFileNameFallback` → `TestLoad_BareFileMissingNameIsSkipped`.

**Files:** `internal/skills/skills.go`, `internal/skills/skills_test.go`

**Risk:** breaks anyone who authored a `.md` skill file without a `name` field.
Mitigation: the error message in `Skipped` tells them exactly what to add.

---

#### Task 0.2: L3 resource indexing (references + scripts + assets)

**Why.** This is the single biggest functional gap. Claude Code skills routinely carry
`references/*.md` (additional docs loaded on demand), `scripts/` (executable
utilities run as black boxes), and `assets/` (templates, fonts, icons). A
code-agent user who installs a Claude Code skill like `pdf` or `pptx` gets the
`SKILL.md` body but cannot access the `scripts/` or `references/` that the body
points at — the skill is effectively broken.

The P6 design doc (§3) already anticipated this: "L3 references are already
implemented — via `read_file`." That's true for references, but the model doesn't
know *where* they are. We need to index them and tell the model.

**What.**

Extend `Skill` to carry resource listings:

```go
type Resource struct {
    Path string // relative to skill root, e.g. "references/api.md"
    Kind string // "reference", "script", "asset"
}

type Skill struct {
    Meta
    Body      string
    Resources []Resource // L3: files the model can read_file or run
    Dir       string     // absolute path to the skill directory, for path resolution
}
```

In `loadDir`, after parsing `SKILL.md`, scan the skill directory for:
- `references/` → `*.md` files → `Kind: "reference"`
- `scripts/` → executable files → `Kind: "script"`
- `assets/` → any files → `Kind: "asset"`

Update `LoadSkillTool.Execute()` to include the resource manifest in the tool
result, after the body:

```
Loaded skill: pptx (v1)

<SKILL.md body>

---
Resources available in this skill:
  references/editing.md     (reference)
  references/pptxgenjs.md   (reference)
  scripts/thumbnail.py      (script)
  scripts/office/unpack.py  (script)
```

The model then uses existing `read_file` / `run_command` with the resolved path.
`LoadSkillTool` must resolve relative resource paths to absolute paths (using
`Skill.Dir`) before returning them, so the model can pass them directly to
`read_file`.

**Files:** `internal/skills/skills.go`, `internal/tools/skill/load_skill.go`,
`internal/skills/skills_test.go`, `internal/tools/skill/load_skill_test.go`

**Non-goal (deferred):** auto-execution convention for `scripts/`. The model
decides to run them; we just tell it they exist. Per-skill scoped tools remain
deferred per P6 §4.

---

#### Task 0.3: Add `license` field to frontmatter

**Why.** Claude Code skills carry a `license` field (e.g. `"Proprietary. LICENSE.txt
has complete terms"` or `"Complete terms in LICENSE.txt"`). code-agent ignores it.
We should at least parse and preserve it — not enforce it, but surface it in the
index so users know the terms before loading.

**What.**
- Add `License string \`yaml:"license"\`` to the `frontmatter` struct.
- Add `License string` to `Meta`.
- Include it in `PromptIndex()` output (append `[license: ...]` or similar terse
  notation).
- Include it in `load_skill` output header.

**Files:** `internal/skills/skills.go`

---

### P1 — Functional enhancements (real user-facing improvement)

#### Task 1.1: Skill resource access tool (`skill_resource` or extend `load_skill`)

**Why.** After Task 0.2, the model knows what resources exist and their absolute
paths. But `read_file` and `run_command` are gated by user permissions. A skill's
`references/` should be **always readable** without a permission prompt — they're
part of the skill the user chose to install. Similarly, a skill's `scripts/` should
be runnable without prompting (the user already trusted the skill by installing it).

**What.**

Option A (preferred): extend `load_skill` to also return resource contents:

```
load_skill(name="pptx", resource="references/editing.md")
→ returns the content of editing.md
```

```
load_skill(name="pptx", resource="scripts/thumbnail.py")
→ returns the script content (model can then decide to run it)
```

This reuses the existing tool and permission model — `load_skill` is already
read-only and auto-approved.

Option B: a separate `skill_resource` tool. More orthogonal but adds a tool name
to the prompt.

Recommendation: **Option A** — fewer tools, less prompt clutter. The `resource`
parameter is optional; omitting it returns the body as today.

**What else:**
- Validate the resource path is within the skill directory (path traversal guard).
- Return the file content directly as the tool result (text for `.md`/`.py`, base64
  for binaries).
- Resource loading emits no separate telemetry event (it's still `load_skill`).

**Files:** `internal/tools/skill/load_skill.go`, `internal/tools/skill/load_skill_test.go`

---

#### Task 1.2: Skill-aware `run_command` path extension

**Why.** When the model sees `scripts/thumbnail.py` in a skill and calls
`run_command` with `python scripts/thumbnail.py`, it expects that to resolve
relative to the skill directory. Currently `run_command` runs from the workspace
root; the model would need to pass the absolute path (which Task 0.2 provides, but
Claude Code skills' own SKILL.md bodies reference relative paths like
`scripts/thumbnail.py`).

**What.**
- When `load_skill` returns, also emit a hint: "Skill scripts are at
  `<absolute-path>/scripts/`. Use absolute paths when running them."
- Alternatively: the model is smart enough to use absolute paths when we give them.
  The resource manifest from Task 0.2 already lists absolute paths. No runtime
  change needed — just verify this works in practice.

**Risk:** low. Defer implementation until we see the model struggle with it.

---

#### Task 1.3: `load_skill` list mode (forgotten-name recovery)

**Why.** When the model forgets a skill name (long conversation, context
truncation), it has no way to re-discover skills without re-reading the system
prompt (which it can't do). A `list_skills` capability was already noted in P6 §12
("Future").

**What.**
- Extend `load_skill` to accept an empty or `"__list__"` name and return the
  current index.
- Or: register a separate `list_skills` tool. But that adds prompt overhead for
  a rarely-used recovery path.

Recommendation: **overload `load_skill`** with the list-on-empty behavior. Simple,
no new tool.

```
load_skill(name="")
→ "Available skills:\n- codeagent-conventions: ...\n- verify-change: ..."
```

**Files:** `internal/tools/skill/load_skill.go`

---

### P2 — Distribution & ecosystem

#### Task 2.1: Plugin marketplace support (install skills from Git repos)

**Why.** code-agent skills today must be manually copied to
`~/.codeagent/skills/` or `<project>/skills/`. Claude Code has a marketplace model:
`/plugin marketplace add <repo>` → `/plugin install <plugin>@<marketplace>`. This
is the biggest UX gap for skill adoption.

**What.**

Phase A — read Claude Code marketplace format:
- Parse `.claude-plugin/marketplace.json` from a local or remote repo.
- Add a `codeagent plugin install <repo-url>` CLI command (or `/plugin install`
  slash command in TUI) that:
  1. Clones/updates the repo to `~/.codeagent/plugins/<marketplace-name>/`
  2. Reads `marketplace.json` to discover plugins
  3. Symlinks or copies the listed skill directories into
     `~/.codeagent/skills/<skill-name>/`
- Add `codeagent plugin list` to show installed plugins.
- Add `codeagent plugin remove <name>`.

Phase B — generate our own `marketplace.json`:
- Create `.claude-plugin/marketplace.json` in the code-agent repo so our seed
  skills can be installed as a Claude Code plugin too (interop in both directions).

**Files:** new `internal/plugins/` package, `cmd/codeagent/plugin.go`

---

#### Task 2.2: `codeagent skill init` scaffolding command

**Why.** Creating a skill from scratch requires knowing the frontmatter format,
directory layout, and conventions. A scaffolding command reduces friction to near
zero.

**What.**
```
codeagent skill init <name>
```
Creates:
```
skills/<name>/
  SKILL.md    # pre-filled frontmatter template with name + empty description
  references/ # empty, with .gitkeep
  scripts/    # empty, with .gitkeep
  assets/     # empty, with .gitkeep
```

The `SKILL.md` template includes commented guidance from P6 §9 (description tips,
gotchas section, keep under 500 lines).

**Files:** `cmd/codeagent/skill.go`, maybe `internal/skills/template.go`

---

### P3 — Quality & observability

#### Task 3.1: Skill hot reload

**Why.** When iterating on a skill (editing `SKILL.md`), you must restart the agent
to pick up changes. The registry is loaded once at startup and never refreshed.
This makes skill authoring painfully slow.

**What.**
- Add `Registry.Reload()` method that re-scans the same directories.
- Expose via a `/reload-skills` slash command in TUI mode, or a `SIGHUP` handler.
- Or simpler: check file mtimes on each `load_skill` call and re-parse if changed.
  Since the registry is read-only and shared, use `sync.RWMutex` for thread safety.

Recommendation: **mtimes on `load_skill`** — minimal change, no new API surface.
On each `Get()`, stat the source file; if mtime > load time, re-parse and update
the in-memory entry. Gate behind a `Development` config flag so production pays
zero overhead.

**Files:** `internal/skills/skills.go`, `internal/tools/skill/load_skill.go`,
`internal/app/config.go`

---

#### Task 3.2: Skill eval harness (correctness regression tests)

**Why.** We have unit tests for parsing and loading, but no behavioral tests that
verify "the model actually follows the skill." Claude Code's `skill-creator` has a
full eval pipeline: test cases → baseline comparison → A/B blind test →
benchmarking. We don't need all of that yet, but we need at least correctness
regression: given a specific prompt, does loading the skill produce the right
behavior?

**What.**
- Add an `internal/skills/eval/` package with a lightweight harness:
  - `evals.json` — test case format: `{prompt, skill, expected_behavior}`
  - `runner_test.go` — loads a fixture registry, calls the model with the prompt,
    asserts the output contains expected markers.
- Start with 3-5 eval cases for `codeagent-conventions`:
  - "add a tool" → model does NOT edit loop.go
  - "commit" → model includes `Co-Authored-By` trailer
  - "debug a test failure" → model fixes source, not test
- Run as part of `go test ./...` with a `-short` skip (requires model API calls).

**Files:** `internal/skills/eval/`, `internal/skills/eval/evals.json`

---

#### Task 3.3: Telemetry — skill trigger rate & source tracking

**Why.** P6 §11 already calls this out: "a skill that is rarely loaded usually has
a bad description." We emit `skill_loaded` events but don't aggregate them. We also
don't track which source (global/project) a loaded skill came from, making it hard
to debug conflicts.

**What.**
- Add `Source string` (`"global"` / `"project"`) to `Skill` and the `skill_loaded`
  telemetry event.
- Add an in-memory counter to `Registry`: `LoadCount(name string) int` — incremented
  on each `AnnounceSkill` call. Expose via a debug endpoint or `/skills` slash
  command.
- Log a warning on startup when a project skill shadows a global skill (name
  collision).

**Files:** `internal/skills/skills.go`, `internal/tools/skill/load_skill.go`,
`internal/agent/event.go`

## 4. What we deliberately defer

Consistent with P6 §4 (non-goals):

- ❌ **Runtime auto-injection.** The model still calls `load_skill` — we never push
  skill bodies into context automatically. This is the spine of P6: "the model owns
  control flow."
- ❌ **Per-skill scoped tools** (`tools:` frontmatter). Deferred per P6 §13.3.
- ❌ **L3 executable convention.** Scripts are discoverable and runnable, but we
  define no formal "this is how scripts declare their interface" convention yet.
- ❌ **Skill-authored-by-agent.** The agent self-authoring skills from its own
  learnings (P6 §12 "Future") remains future work.
- ❌ **Skill dependencies** (skill A requires skill B). Not needed at current scale.

## 5. Success criteria

After P0 + P1 are done, a user should be able to:

1. Clone `anthropic-agent-skills` and install any skill into code-agent.
2. `load_skill("pptx")` returns the body **and** the list of scripts/references.
3. The model can `read_file` a skill's `references/editing.md` without guessing the path.
4. A skill without a `name` in frontmatter is rejected with a clear error message.
5. The `license` field appears in the skill index.

After P2 is done:

6. `codeagent plugin install https://github.com/anthropics/skills` installs all
   skills from the marketplace.
7. `codeagent skill init my-skill` scaffolds a valid skill directory.

After P3 is done:

8. Editing `SKILL.md` takes effect on the next `load_skill` call without restart.
9. `go test ./internal/skills/eval/...` runs behavioral correctness checks.
10. A rarely-loaded skill is surfaced via telemetry (under-triggering → bad description).

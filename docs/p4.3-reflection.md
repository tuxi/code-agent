# P4.3 — Reflection (design PRD)

> Status: **P4.3.a/b/c shipped (§12), now under revision — see §P4.3-R.**
> Like P4.1, it defines a **data + nudge** layer, not a control machine — see
> "Guardrail". Companion to [docs/p4.1-observation.md](./p4.1-observation.md).
>
> ⚠️ **§P4.3-R (2026-07-04) supersedes parts of §5, §9, and §13.** The shipped
> design put a *mechanical quality-signal extractor* in the runtime; field use
> showed that orientation is misaligned with the best-in-class design (Claude
> Code). Read §1–§13 as the historical record of what shipped; read **§P4.3-R**
> for the finalized forward direction.

## 1. Background

P4.1 made failures *legible*: the model now sees `failure=test`, the salient
diagnostics, and acts on them. In a real run that produced exactly the loop we
wanted — `go test` failed → the model read the code → edited → re-ran → green.

But that same run exposed the next gap. The failing test was
`TestLoadConfigMultiModel` (expecting `compact_ratio` default `0.7`, got `0.5`).
The model "fixed" it by editing **`config_test.go` — the test itself** — and the
suite went green. Whether that was right depends on intent, but the agent had
**no check on fix quality**: it cannot tell "I fixed the cause" from "I changed
the test to accept the wrong value." This is the classic paper-over / reward-hack
failure mode.

P4.1 answers *"what happened."* P4.3 answers *"did I actually fix it, or paper
over it?"* — and *"did I leave anything unverified?"*

## 2. Goal

At the moment the model decides it is **done**, give it one grounded chance to
check its own work against what actually happened this turn, so it can catch a
paper-over or an unverified claim and continue if needed — before the user sees
a false "done."

```
model: (no tool calls) → wants to finish
        ↓  reflection: build a self-check from THIS TURN's real Observations
[reflection] you edited a test file after it failed, then it passed —
             did you fix the cause or the test? + anything unverified?
        ↓  model re-judges: continue (fix properly) OR confirm done
```

## 3. The mechanism already exists (this is the guardrail)

The loop already injects **ephemeral, non-persisted nudges** that steer the model
without taking control from it:

- `withConvergenceNudge` — when a turn has made many tool calls, append a
  transient `[reminder]` that steers toward answering. *"appended only to this
  request, never persisted … keeps applying pressure without polluting history.
  (max_steps still backstops it.)"*
- `finalAnswerAfterLimit` — at the step limit, one tool-free call with an
  ephemeral nudge to extract a final answer.

**Reflection is the mirror image of the convergence nudge.** The convergence
nudge fires when the model *over*-works ("you've done enough, stop"). Reflection
fires when the model is about to *under*-verify ("before you stop, check this").
Same machinery: an ephemeral, non-persisted nudge whose *content* is grounded in
real turn data; the model decides what to do with it. It is **not** a new control
paradigm and **not** a separate reasoning engine — it is the existing nudge
pattern fired at the finalize boundary.

## 4. Guardrail

Reflection produces **data** (a `ReflectionContext`) and a **question** (an
ephemeral nudge). It must never:

- ❌ decide to retry/fix on its own — it asks; the model acts.
- ❌ hardcode verdicts like "a test file was edited ⇒ reject the answer."
- ❌ loop: at most **one** reflection nudge per turn (a `reflected` flag), so it
  cannot ping-pong. `max_steps` remains the backstop.
- ❌ persist the nudge into history (ephemeral, like the convergence nudge).
- ❌ become a heavyweight reflection subsystem. It is signal extraction + a
  prompt builder + a one-line loop hook.

The runtime decides only **whether to ask** (based on factual signals, exactly
as the convergence nudge decides based on tool-call count). The model decides
**everything about the outcome**.

## 5. Non-goals

- ❌ Judging correctness of code. Reflection surfaces facts ("you edited a test
  after it failed"); the model judges.
- ❌ A scoring/grading model or a second "critic" agent. *(§P4.3-R keeps this —
  no in-loop critic — but permits a **deterministic** verify hook that runs the
  real build/test, and permits **model self-judgment**, in place of the runtime
  guessing quality.)*
- ❌ Blocking the final answer for more than one reflection pass.
- ❌ Firing on turns with nothing to verify (pure Q&A or read-only turns).
  *(VIOLATED in practice: a pure-doc-write turn has nothing verifiable yet fires
  — see §P4.3-R Move 1. The doc false-fire is a regression against this very
  bullet, not a new policy.)*

## 6. Data model

```go
package reflection

// ReflectionContext is the factual, machine-extracted state of a turn's work,
// derived from the loop's Steps and their Observations. It is pure data — the
// raw material the self-check prompt is built from. No judgments, no advice.
type ReflectionContext struct {
    MutatedFiles      []string // files changed via edit_file / apply_patch this turn
    TestFilesMutated  []string // subset of MutatedFiles matching a test pattern (_test.go, …)
    LastVerifyPassed  *bool    // result of the turn's most recent build/test (nil = none ran)
    LastVerifyScope   string   // e.g. "./..." (full) vs a narrower path, or "" if none
    TestEditedAfterFailure bool // a test file was edited following a failure=test observation
    UnverifiedMutation     bool // code changed, but no passing verify ran afterward
    FailuresSeen      []string // failure types observed this turn (compile/test/…)
}

// Concerns reports whether anything is worth asking the model about. When false,
// the loop accepts the final answer immediately (no nudge, no extra model call).
func (c ReflectionContext) Concerns() bool
```

## 7. Trigger & loop integration

The loop already has the finalize boundary:

```go
if !resp.HasToolCalls() {
    turn.Final = resp.Content
    return turn, nil
}
```

Reflection slots in there, mirroring the convergence-nudge shape:

```go
if !resp.HasToolCalls() {
    if r.Reflector != nil && !reflected {
        rc := r.Reflector.Reflect(turn.Steps)   // pure extraction
        if rc.Concerns() {
            reflected = true                      // strictly one-shot
            sess.Messages = append(sess.Messages, resp.AssistantMessage())
            // ephemeral nudge: appended for the NEXT request only, not persisted
            nudge = reflection.Nudge(rc)
            r.emit(Event{Kind: EventReflected, ...})
            continue                              // give the model one more pass
        }
    }
    turn.Final = resp.Content
    return turn, nil
}
```

Notes:
- The nudge is applied the same ephemeral way as `withConvergenceNudge`
  (appended to the request `msgs`, not persisted to `sess.Messages`).
- `reflected` is a per-turn bool: one reflection pass, ever. If the model
  finalizes again, that answer is accepted.
- A `Reflector` interface lives in `agent` (like `Approver` / `Observer`); the
  implementation lives in `reflection`. Nil-safe: unset ⇒ today's behavior.

```go
// in agent
type Reflector interface {
    Reflect(steps []Step) reflection.ReflectionContext
}
```

## 8. The reflection nudge (content)

`reflection.Nudge(rc)` renders an ephemeral `[reflection]` message surfacing
**only the triggered signals**, as questions, grounded in the real turn:

```
[reflection] Before you finish, check your work against what actually happened:
- You edited a test file (internal/app/config_test.go) after a test failure,
  then the suite passed. Did you fix the root cause, or change the test to
  accept a wrong value? If the test was correct, fix the source instead.
- You changed internal/app/config.go but no build/test has confirmed it since.
  Run a verification before claiming done.
If everything genuinely checks out, give your final answer now. Otherwise, fix it.
```

Only the lines whose signal fired are included. If `Concerns()` is false, no
nudge is produced and the answer stands.

## 9. Signal extraction (from Steps + Observations)

All pure, derived from `turn.Steps` (each carries `ToolName`, `ToolInput`,
`Observation`). P4.1's `Observe` already classifies tool results, so reflection
reuses that signal rather than re-parsing:

- **MutatedFiles** — `edit_file` / `apply_patch` steps → their target paths.
- **TestFilesMutated** — MutatedFiles matching `*_test.go` (Go first; extensible).
- **LastVerifyPassed / Scope** — the last `run_command` step whose command is a
  build/test; pass = its Observation is `ok`; scope from the command (`./...`
  vs a path).
- **TestEditedAfterFailure** — a test-file edit whose step index is *after* a
  step with a `failure=test` observation.
- **UnverifiedMutation** — a mutation step exists with no passing build/test
  step after it.
- **FailuresSeen** — union of failure types across the turn's observations.

## 10. Testing strategy

Offline, table-driven over synthetic `[]Step` sequences — no model:

- `extract_test.go`: step sequences → expected `ReflectionContext` (the
  config_test.go transcript is a fixture: edit-test-after-fail ⇒ true).
- `nudge_test.go`: `ReflectionContext` → the nudge contains exactly the
  triggered lines and omits the rest; `Concerns()==false` ⇒ empty.
- A loop test (fake model) proving: (a) a paper-over turn triggers exactly one
  reflection pass and a continue; (b) a clean turn triggers none; (c)
  `Reflector==nil` is byte-for-byte today's behavior; (d) one-shot — never two
  reflection passes in a turn.

## 11. How it stays emergent (relationship to P4.2)

P4.2's verify-fix loop already emerges from P4.1. P4.3 does not add control to it
— it adds a *grounded question at the finish line*. If the model agrees it
paper-ed over, it continues and the verify-fix loop runs again, on its own. If it
disagrees, it answers. The runtime never forces a fix. This is still the Phase 4
"Done when: the model … fixes them **without any hardcoded sequencing**" —
reflection just makes the model less likely to stop one step too early.

## 12. Phasing

- **P4.3.a** — ✅ **shipped.** `reflection` package: `StepView`,
  `ReflectionContext`, the pure extractor, `Concerns()`, `Nudge()`. No loop
  change, fully table-tested (incl. the real config_test.go paper-over fixture).
- **P4.3.b** — ✅ **shipped.** `agent.Reflector` interface + `DefaultReflector`
  (adapts `[]Step` → `[]StepView`) + `Runner.Reflector` injection + the one-shot
  finalize-boundary hook (re-prompt with an ephemeral nudge; a premature finish
  is never persisted) + `EventReflected`. Wired into both runners. Loop tests
  cover: one-shot fire + continue, clean turn skips, nil-Reflector is today's
  behavior, premature "done" not persisted.
- **P4.3.c** — ✅ **shipped.** A concise `[reflection] …` console line printed
  when the self-check fires, so the human sees *why* the agent took another pass.

## 13. Decisions (signed off)

1. **Active vs passive** — ✅ **active, strictly one-shot.** Inject the nudge and
   give the model one more pass; passive would warn the human but never improve
   the agent. One-shot (`reflected` flag) keeps it safe.
2. **One-shot vs budget** — ✅ **one reflection pass per turn.** Raise only if
   telemetry shows real issues left on the table.
3. **Default on or off** — ✅ **on.** The trigger is conditional (`Concerns()`),
   so a clean turn pays nothing (no nudge, no extra model call).
4. **First signals** — ✅ **`TestEditedAfterFailure` + `UnverifiedMutation` only.**
   They cover the observed failure mode. Partial-verification / scope and
   `FailuresSeen` are deferred to telemetry; the §6 struct ships this subset in
   P4.3.a. *(SUPERSEDED by §P4.3-R: `UnverifiedMutation` is demoted — it is a
   mechanical **guess** that verification "should" have run, which is exactly the
   orientation being retired. `TestEditedAfterFailure` survives because it reports
   a **fact** ("you edited a test after it failed"), not a quality verdict.)*
5. **Test-file pattern** — ✅ **Go `*_test.go` only**, matching where toolchain
   support actually is (same discipline as P4.1).

# P4.3-R — Revision: pivot the quality gate toward Claude Code

> Status: **finalized direction (2026-07-04).** Design, not yet implemented.
> Supersedes the parts of §5, §9, §13 flagged above. Grounded in the current
> codebase, not aspiration: every mechanism it leans on already exists.

## R.1 Why revise now

The original PRD deferred two options as premature because the machinery didn't
exist yet. It exists now, so the timing objection is gone:

- **Plan mode** — a real pre-action vehicle: [docs/p10.1-plan-mode.md](./p10.1-plan-mode.md),
  `internal/agent/plan.go`, the ephemeral `planningPrompt` injected at
  `internal/agent/loop.go` (the same first-call injection point as the skills /
  parallel reminders).
- **Hypothesis-first directive** — already in the system prompt
  (`internal/prompt/system.go`, "Debugging — say your hypothesis BEFORE the deep
  dive"). Stated, but nothing reinforces it mid-flight.
- Thinking, subagents (`verify`, code-review) — the "reason before/after acting"
  and "critic in a fresh context" pieces Claude Code relies on.

Two field failures forced the decision:

- **投机取巧 / green-build shortcut** (SwiftUI: keyvalue-count-as-line-count →
  `GeometryReader` + `PreferenceKey`). Reflection **never fired**: the build was
  green and no test file changed, so `Concerns() == false`. The real defect was
  **pre-action** — the model skipped its own hypothesis-first rule. A post-hoc,
  finalize-boundary check structurally cannot reach a pre-action defect.
- **False-fire on pure docs**: 5× `create_file` on `.md` → `UnverifiedMutation`
  → the nudge tells the model to "run a verification" on markdown; the model
  obediently burns a step. A mechanical file-diff heuristic is low-precision.

**Diagnosis.** The misaligned part is *not* the ephemeral-nudge mechanism, and
*not* the finalize hook point — those match Claude Code's `<system-reminder>` and
`Stop` hook respectively. The misaligned part is the **orientation**: the runtime
mechanically extracting "quality signals" from file-path diffs and *guessing*
whether verification ran. That is simultaneously **low-recall** (misses
green-build shortcuts) and **low-precision** (false-fires on docs). Claude Code
deliberately does **not** do this — it puts quality gates in **deterministic
hooks** (run the real tests) or the **model's own reasoning** (plan mode,
thinking), and keeps the runtime out of the "judge code quality" business.

## R.2 What is KEPT (validated by the Claude Code mirror)

- The **ephemeral, non-persisted nudge** mechanism (`appendEphemeralUser`) —
  Claude Code's `<system-reminder>` analog. Keep.
- The **finalize-boundary hook point** — Claude Code's `Stop` / `SubagentStop`
  analog. Keep the *point*; change *what happens there*.
- **One-shot, model-owns-outcome, nil-safe.** Keep.

## R.3 What changes — three moves

### Move 1 — Bug fix, ship first, direction-independent

Stop false-firing on non-code. This honors §5's own "nothing to verify" non-goal.

- Add `isVerifiableCode(path)` (mirror of the existing `isTestFile` precedent in
  `internal/reflection/reflection.go`). Doc/data files (`.md/.txt/.rst/.json/
  .yaml/.toml/.csv/…`) are not verifiable code.
- Gate `UnverifiedMutation` on **≥1 verifiable-code file** among the turn's
  mutations, **and** filter the nudge's file listing to verifiable-code files
  only — both the trigger (`reflection.go`) **and** the listing (`nudge.go` joins
  *all* `MutatedFiles` today).
- Mixed turn (code + doc) still warns, but lists only the code files.
- Cost: one predicate + two call sites, zero risk. Closes the doc false-fire
  regardless of the larger pivot.

### Move 2 — Direction pivot: the finish-line gate stops *guessing*

`UnverifiedMutation` is retired **as a quality claim**. The runtime must not
assert "no build/test confirmed it" from tool-call patterns. Replace with one of:

- **(2a) Deterministic verify hook** — the true port of Claude Code's `Stop`
  hook. At finalize, if the turn mutated verifiable code, run the project's real
  verify command (we already have `isVerifiableCode` + `run_command` +
  P4.1 `[observation]` markers) and feed the **real** result back — truth, not a
  guess.
- **(2b) Model self-judgment** — drop the mechanical signal entirely; rely on
  hypothesis-first + plan mode + thinking for the model to verify on its own.

`TestEditedAfterFailure` is **retained either way**: it reports a **fact** ("you
edited a test after it failed"), phrased as a question — the "state facts, model
judges" line the original §5 already endorsed. It is not the runtime judging
quality.

**Finalized default: 2a for turns that mutated verifiable code, falling back to
2b's fact-only nudge when no verify command is configured.** 2a gives real signal
instead of a guess and is the closest match to the best-in-class design; confirm
this at implementation kickoff.

### Move 3 — Direction pivot: move the self-check *earlier*

The shortcut failure is a pre-action defect, so put a check before the first
mutation — the mirror image the original §3 described, fired at the **start** line
instead of the finish line.

- The hypothesis-first directive already exists (`system.go`) but a model ignored
  it. Reinforce it with an **ephemeral pre-mutation nudge** — same first-call
  injection mechanism as `skillsReminder` / `parallelReminder` — fired **before
  the first project mutation** on turns that look like diagnosis / a repeated
  attempt / a non-obvious fix, reminding the model to state its **root-cause
  hypothesis** first.
- This directly targets 投机取巧: shortcuts happen because the model skips "think
  about the root cause" — precisely the failure the model self-diagnosed in the
  SwiftUI case. No new subsystem: it reuses the existing injection point and the
  existing directive.

> **Shipped trigger (precision decision).** Of the three candidate turn shapes,
> only **"a repeated attempt"** is a clean, high-precision *runtime* signal, so
> the shipped check fires exactly there: when the model is **about to make a
> mutating call AND a `failure=…` observation has already surfaced this turn**
> (the paper-over hot zone). It does **not** try to classify "non-obvious fix"
> from the prompt — that would be a brittle guess and would tax every edit with an
> extra round trip. The mechanism is the finalize self-check mirrored at the
> mutation boundary: the premature edit is **dropped** (not persisted, not
> executed) and re-prompted with the ephemeral `hypothesisReminder`; one-shot per
> turn (`hypothesized`); opt-in via `RemindHypothesis` (on in the main runner,
> off for the read-only subagent, which cannot mutate). **Known gap:** a
> *first-attempt* shortcut with no prior tool failure (the SwiftUI first edit) is
> not caught by runtime signal — that case stays with the `system.go` directive.
> Raise coverage only if telemetry shows it left value on the table.

## R.4 Guardrail (unchanged in spirit)

- The runtime decides only **whether to ask**, or **whether to run real
  verification**. The model owns every outcome.
- **No** scoring model, **no** in-loop critic agent, **no** hardcoded verdicts.
  (2a runs a deterministic command and reports truth — that is not a verdict.)
- One-shot per boundary; ephemeral; nil-safe; a clean turn pays nothing.

## R.5 Phasing

- **R.1** — Move 1 (doc false-fire fix). ✅ **shipped.** `isVerifiableCode` /
  `isDocOrData` predicates + `CodeFilesMutated`; `UnverifiedMutation` gates on a
  verifiable-code mutation; the nudge lists only code files. Table tests: doc-only
  turn is silent, mixed turn warns on code only.
- **R.2** — Move 3 (pre-mutation hypothesis nudge). ✅ **shipped.**
  `reflection.IsMutatingTool` + `SawFailure`; loop-local `RemindHypothesis` gate +
  one-shot `hypothesized` drop-and-reprompt at the mutation boundary +
  `hypothesisReminder` + `EventPreMutation`; wired on in the main runner. Loop
  tests: fires once before an edit after a failure, silent without a failure,
  opt-in off. See the shipped-trigger note under Move 3 for the precision decision.
- **R.3** — Move 2 (retire `UnverifiedMutation` as a quality claim; add the 2a
  deterministic verify hook with the 2b fallback). ✅ **shipped.** New
  `agent.verify_command` config → `Runner.VerifyCommand`; at finalize, an
  unverified verifiable-code change with a configured command runs
  `runFinalizeVerify` — a runtime action (not a fabricated model turn) that
  executes the real command via the existing tool executor + Observer and feeds
  the **truth** back: a pass accepts the finish, a failure re-prompts with the
  real error (`EventVerified` + a fact-based `EventReflected`). With **no**
  command (2b) the unverified case is **silent** — `reflection.Nudge()` no longer
  renders the retired "run a verification" guess; only `TestEditedAfterFailure`
  (a fact) still nudges. Safety guards: skips when `run_command` is absent or the
  command would mutate (`HasSideEffectsFor`); one-shot per turn (`verified`).
  Loop tests: pass accepts, fail re-prompts once then accepts the fix, no-command
  is silent.

> **Config home is provisional.** `agent.verify_command` was placed in
> `config.yaml` expediently. Project *behavior* belongs in a project-local
> settings file, not the infra config — see [docs/p11-project-settings.md](./p11-project-settings.md),
> which relocates `verify` into `.codeagent/settings.json` / `settings.local.json`
> (Claude-style) with auto-detection and agent self-write. `agent.verify_command`
> stays as the back-compat / embedded-host layer.

> **2a-vs-2b decision (closed).** The doc left the finish-line gate as 2a-default
> with a 2b fallback. Shipped exactly that: **2a when `verify_command` is
> configured, 2b (silence) otherwise.** 2a is implemented as a *runtime-run*
> verify rather than a fabricated model tool-call — running it through the real
> executor keeps determinism (Claude Code's Stop-hook property) while the model
> still owns the *fix*, so the "model owns control flow" invariant holds: the
> runtime reports truth, the model decides what to do about a failure.

## R.6 One-line conclusion

Keep reflection's **mechanism** (ephemeral nudge) and its **hook points** (start
+ finish line); retire the **orientation** of the runtime mechanically guessing
code quality. Push self-correction to **before** the work (hypothesis-first) and
make the finish-line gate **run real verification** rather than infer it — exactly
where Claude Code puts it.

# P4.3 — Reflection (design PRD)

> Status: design. No implementation yet. This is the spec to hand to a developer.
> Like P4.1, it defines a **data + nudge** layer, not a control machine — see
> "Guardrail". Companion to [docs/p4.1-observation.md](./p4.1-observation.md).

## 1. Background

P4.1 made failures *legible*: the model now sees `failure=test`, the salient
diagnostics, and acts on them. In a real run that produced exactly the loop we
wanted — `go test` failed → the model read the code → edited → re-ran → green.

But that same run exposed the next gap. The failing test was
`TestLoadConfigMultiModel` (expecting `compact_ratio` default `0.7`, got `0.5`).
The model "fixed" it by editing **`config_test.go` — the test itself** — and the
suite went green. Whether that was right depends on intent, but the agent had
**no check on fix quality**: it cannot tell "I fixed the cause" from "I changed
the test to accept the wrong value." This is the classic paper-over / reward-hack
failure mode.

P4.1 answers *"what happened."* P4.3 answers *"did I actually fix it, or paper
over it?"* — and *"did I leave anything unverified?"*

## 2. Goal

At the moment the model decides it is **done**, give it one grounded chance to
check its own work against what actually happened this turn, so it can catch a
paper-over or an unverified claim and continue if needed — before the user sees
a false "done."

```
model: (no tool calls) → wants to finish
        ↓  reflection: build a self-check from THIS TURN's real Observations
[reflection] you edited a test file after it failed, then it passed —
             did you fix the cause or the test? + anything unverified?
        ↓  model re-judges: continue (fix properly) OR confirm done
```

## 3. The mechanism already exists (this is the guardrail)

The loop already injects **ephemeral, non-persisted nudges** that steer the model
without taking control from it:

- `withConvergenceNudge` — when a turn has made many tool calls, append a
  transient `[reminder]` that steers toward answering. *"appended only to this
  request, never persisted … keeps applying pressure without polluting history.
  (max_steps still backstops it.)"*
- `finalAnswerAfterLimit` — at the step limit, one tool-free call with an
  ephemeral nudge to extract a final answer.

**Reflection is the mirror image of the convergence nudge.** The convergence
nudge fires when the model *over*-works ("you've done enough, stop"). Reflection
fires when the model is about to *under*-verify ("before you stop, check this").
Same machinery: an ephemeral, non-persisted nudge whose *content* is grounded in
real turn data; the model decides what to do with it. It is **not** a new control
paradigm and **not** a separate reasoning engine — it is the existing nudge
pattern fired at the finalize boundary.

## 4. Guardrail

Reflection produces **data** (a `ReflectionContext`) and a **question** (an
ephemeral nudge). It must never:

- ❌ decide to retry/fix on its own — it asks; the model acts.
- ❌ hardcode verdicts like "a test file was edited ⇒ reject the answer."
- ❌ loop: at most **one** reflection nudge per turn (a `reflected` flag), so it
  cannot ping-pong. `max_steps` remains the backstop.
- ❌ persist the nudge into history (ephemeral, like the convergence nudge).
- ❌ become a heavyweight reflection subsystem. It is signal extraction + a
  prompt builder + a one-line loop hook.

The runtime decides only **whether to ask** (based on factual signals, exactly
as the convergence nudge decides based on tool-call count). The model decides
**everything about the outcome**.

## 5. Non-goals

- ❌ Judging correctness of code. Reflection surfaces facts ("you edited a test
  after it failed"); the model judges.
- ❌ A scoring/grading model or a second "critic" agent.
- ❌ Blocking the final answer for more than one reflection pass.
- ❌ Firing on turns with nothing to verify (pure Q&A or read-only turns).

## 6. Data model

```go
package reflection

// ReflectionContext is the factual, machine-extracted state of a turn's work,
// derived from the loop's Steps and their Observations. It is pure data — the
// raw material the self-check prompt is built from. No judgments, no advice.
type ReflectionContext struct {
    MutatedFiles      []string // files changed via edit_file / apply_patch this turn
    TestFilesMutated  []string // subset of MutatedFiles matching a test pattern (_test.go, …)
    LastVerifyPassed  *bool    // result of the turn's most recent build/test (nil = none ran)
    LastVerifyScope   string   // e.g. "./..." (full) vs a narrower path, or "" if none
    TestEditedAfterFailure bool // a test file was edited following a failure=test observation
    UnverifiedMutation     bool // code changed, but no passing verify ran afterward
    FailuresSeen      []string // failure types observed this turn (compile/test/…)
}

// Concerns reports whether anything is worth asking the model about. When false,
// the loop accepts the final answer immediately (no nudge, no extra model call).
func (c ReflectionContext) Concerns() bool
```

## 7. Trigger & loop integration

The loop already has the finalize boundary:

```go
if !resp.HasToolCalls() {
    turn.Final = resp.Content
    return turn, nil
}
```

Reflection slots in there, mirroring the convergence-nudge shape:

```go
if !resp.HasToolCalls() {
    if r.Reflector != nil && !reflected {
        rc := r.Reflector.Reflect(turn.Steps)   // pure extraction
        if rc.Concerns() {
            reflected = true                      // strictly one-shot
            sess.Messages = append(sess.Messages, resp.AssistantMessage())
            // ephemeral nudge: appended for the NEXT request only, not persisted
            nudge = reflection.Nudge(rc)
            r.emit(Event{Kind: EventReflected, ...})
            continue                              // give the model one more pass
        }
    }
    turn.Final = resp.Content
    return turn, nil
}
```

Notes:
- The nudge is applied the same ephemeral way as `withConvergenceNudge`
  (appended to the request `msgs`, not persisted to `sess.Messages`).
- `reflected` is a per-turn bool: one reflection pass, ever. If the model
  finalizes again, that answer is accepted.
- A `Reflector` interface lives in `agent` (like `Approver` / `Observer`); the
  implementation lives in `reflection`. Nil-safe: unset ⇒ today's behavior.

```go
// in agent
type Reflector interface {
    Reflect(steps []Step) reflection.ReflectionContext
}
```

## 8. The reflection nudge (content)

`reflection.Nudge(rc)` renders an ephemeral `[reflection]` message surfacing
**only the triggered signals**, as questions, grounded in the real turn:

```
[reflection] Before you finish, check your work against what actually happened:
- You edited a test file (internal/app/config_test.go) after a test failure,
  then the suite passed. Did you fix the root cause, or change the test to
  accept a wrong value? If the test was correct, fix the source instead.
- You changed internal/app/config.go but no build/test has confirmed it since.
  Run a verification before claiming done.
If everything genuinely checks out, give your final answer now. Otherwise, fix it.
```

Only the lines whose signal fired are included. If `Concerns()` is false, no
nudge is produced and the answer stands.

## 9. Signal extraction (from Steps + Observations)

All pure, derived from `turn.Steps` (each carries `ToolName`, `ToolInput`,
`Observation`). P4.1's `Observe` already classifies tool results, so reflection
reuses that signal rather than re-parsing:

- **MutatedFiles** — `edit_file` / `apply_patch` steps → their target paths.
- **TestFilesMutated** — MutatedFiles matching `*_test.go` (Go first; extensible).
- **LastVerifyPassed / Scope** — the last `run_command` step whose command is a
  build/test; pass = its Observation is `ok`; scope from the command (`./...`
  vs a path).
- **TestEditedAfterFailure** — a test-file edit whose step index is *after* a
  step with a `failure=test` observation.
- **UnverifiedMutation** — a mutation step exists with no passing build/test
  step after it.
- **FailuresSeen** — union of failure types across the turn's observations.

## 10. Testing strategy

Offline, table-driven over synthetic `[]Step` sequences — no model:

- `extract_test.go`: step sequences → expected `ReflectionContext` (the
  config_test.go transcript is a fixture: edit-test-after-fail ⇒ true).
- `nudge_test.go`: `ReflectionContext` → the nudge contains exactly the
  triggered lines and omits the rest; `Concerns()==false` ⇒ empty.
- A loop test (fake model) proving: (a) a paper-over turn triggers exactly one
  reflection pass and a continue; (b) a clean turn triggers none; (c)
  `Reflector==nil` is byte-for-byte today's behavior; (d) one-shot — never two
  reflection passes in a turn.

## 11. How it stays emergent (relationship to P4.2)

P4.2's verify-fix loop already emerges from P4.1. P4.3 does not add control to it
— it adds a *grounded question at the finish line*. If the model agrees it
paper-ed over, it continues and the verify-fix loop runs again, on its own. If it
disagrees, it answers. The runtime never forces a fix. This is still the Phase 4
"Done when: the model … fixes them **without any hardcoded sequencing**" —
reflection just makes the model less likely to stop one step too early.

## 12. Phasing

- **P4.3.a** — `reflection` package: `ReflectionContext`, the pure extractor over
  `[]Step`, `Concerns()`, and `Nudge()`. No loop change. Fully table-tested.
- **P4.3.b** — `agent.Reflector` interface + `Runner.Reflector` injection + the
  one-shot finalize-boundary hook + `EventReflected`. Wire a default reflector.
- **P4.3.c** — UI: a concise `[reflection] …` console line (it is ephemeral, so
  surfacing it helps the human see *why* the agent did another pass).

## 13. Decisions (signed off)

1. **Active vs passive** — ✅ **active, strictly one-shot.** Inject the nudge and
   give the model one more pass; passive would warn the human but never improve
   the agent. One-shot (`reflected` flag) keeps it safe.
2. **One-shot vs budget** — ✅ **one reflection pass per turn.** Raise only if
   telemetry shows real issues left on the table.
3. **Default on or off** — ✅ **on.** The trigger is conditional (`Concerns()`),
   so a clean turn pays nothing (no nudge, no extra model call).
4. **First signals** — ✅ **`TestEditedAfterFailure` + `UnverifiedMutation` only.**
   They cover the observed failure mode. Partial-verification / scope and
   `FailuresSeen` are deferred to telemetry; the §6 struct ships this subset in
   P4.3.a.
5. **Test-file pattern** — ✅ **Go `*_test.go` only**, matching where toolchain
   support actually is (same discipline as P4.1).

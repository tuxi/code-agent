# code-agent auto mode 功能需求文档

> 版本：v0.2（代码核对后锁定 Phase 1 契约，见 §12）
> 目标读者：负责实现 auto mode 的开发
> 关系：auto mode 是 `/goal` 无人值守路径的**上游前置**，但两者是独立工作流，可并行开发，只在末端「注入哪个 Approver」处汇合。本文档可与 `code-agent-goal-需求文档.md` 平行推进。

---

## 1. 背景与要解决的问题

code-agent 的 agent loop 在执行有副作用的工具调用（写文件、跑命令）前，会经过 `Approver` 这道关。当前唯一实现是交互式的 `ConfirmApprover`，对每个副作用动作弹一个 y/N。

这带来两个问题：
- 任何较长的自主任务（尤其 `/goal` 的连续多轮推进）都会被一连串 y/N 拴住，拿不到「走开、回来再看结果」的体验。
- 这不是 `/goal` 独有的痛点——任何 agent session 跑长任务都受此影响。

auto mode 要解决的是：在**有明确安全边界**的前提下，让 agent 自主放行那些**可逆、限定在工作区内**的副作用动作，而把不可逆、越界、危险的动作**始终**留给人确认。

一句话定义：auto mode 是 `Approver` 接口的一个**策略驱动实现**，按「可逆 + 限定爆炸半径」的分级自动放行，默认关闭、显式开启。

**核心认知（务必对齐）**：auto mode 的真正工作量**不在「移除那个确认提示」**，而在**审批策略本身**——哪些能自动放行、哪些永远要人确认、以及如何在 agent 会读取不可信内容的前提下保证这条边界不被绕过。把 auto mode 理解成「无脑放行一切」是危险的，也正是它必须作为独立、认真设计的全局关注点的原因。

---

## 2. 范围

### 2.1 在范围内
- 一个实现现有 `Approver` 接口的策略驱动审批器（暂称 `AutoApprover`）。
- 开关与生命周期：REPL 命令 `/auto on|off`、headless/CI 标志（如 `--auto`）、默认关闭。
- 审批分级策略（本文档 §4，核心）。
- 工作区爆炸半径限定与命令允许/拒绝清单。
- 每次自动放行的审计事件。
- 即时停机（kill switch）。

### 2.2 明确不在范围内
- `/goal` 的循环本身（在 goal 包，单独交付）。
- 工具的具体实现与 sandbox 的执行隔离（auto mode 只管「批不批」，不管「怎么执行」）。
- 把审批策略塞进 goal 包（审批是全局关注点，必须独立）。

---

## 3. 接口缝（载重的解耦点）

auto mode 是现有 `Approver` 接口的**drop-in 实现**，**不改 agent loop 的调用路径**：

- loop 现状：对 `tools.HasSideEffectsFor(tool, input)` 为真的调用才咨询 `r.Approver`；只读工具从不咨询；`Approver` 为 nil 时**拒绝**（fail-safe deny）。
- auto mode 落地 = 把 `agent.Runner.Approver` 从 `ConfirmApprover` 换成 `AutoApprover`，接口签名不变（接收工具名 + 调用参数，返回放行与否）。
- 与 `/goal` 的汇合点**仅此一处**：goal 的 worker 原样继承 `Runner.Approver`，goal 包对 auto mode 零感知。

复用、不重造（与本项目既有做法一致）：
- 复用 `tools.HasSideEffectsFor` 做「是否副作用」的判定，auto mode 的分级策略**叠在它之上**，不另起一套副作用分类。
- 复用工具的参数结构体做路径/命令的论据检查，不正则瞎解析。

---

## 4. 审批分级策略（核心）

判定的输入是 **(工具名, 调用参数)**——不是只看工具，要看论据。分四级：

| 级别 | 内容 | auto mode 行为 |
|---|---|---|
| 只读 | read_file / grep / 等无副作用工具 | 从不咨询审批（现状不变） |
| 工作区内可逆写 | edit_file 等，且目标路径经规范化后确在 workspace root 内 | **自动放行** |
| 受控命令 | `run_command`，且命令命中**用户配置的允许清单**（项目的 test/build/format 命令） | **自动放行**；未命中 → 转人工确认 |
| 永不自动（始终确认） | 见下「never-auto 清单」 | **始终人工确认**，不受 auto mode、配置、`/goal` 或任何指令影响 |

**never-auto 清单（硬规则，不可被任何配置/指令覆盖）：**
- 不可逆/破坏性：删除工作区外文件、`rm -rf`、清空目录、覆盖式批量操作。
- 越界：任何作用于 workspace root 之外路径的动作。
- 版本控制高危：`git push`、force-push、改写历史（rebase/reset --hard 到远端、filter-branch 等）。
- 提权与系统级：`sudo`、改系统/全局配置、安装全局依赖。
- 网络与外部投放：任意对外网络（`curl|sh`、下载执行、向外部 endpoint 发数据）。
- 凭据/密钥：读取或写入凭据、token、密钥文件。

**两条压倒性规则：**
1. **拒绝清单优先**：即便某命令命中了允许清单，只要触碰 never-auto 特征（含 shell 元字符可能逃逸、管道、`&&`、子 shell、环境变量改写、路径逃逸 `..`），一律转人工确认。命令分类必须**保守**——任何无法完全识别为「简单且安全」的命令，转人工。
2. **不确定即确认（fail-safe to human）**：策略无法自信地把一个动作归类为安全时，默认转人工确认，**绝不默认自动放行**（与 loop 现有「nil Approver → deny」同向）。

> **⚠️ 代码核对修订（v0.2）**：上表「受控命令」一行的心智模型已被代码事实修正——**Approver 永远只看到 `sandbox.Classify==Confirm` 的命令**；安全的 test/build/format 命令早在 `Allow` 档自动跑、根本不到 Approver。因此 Phase 1 的命令侧实际策略、never-auto↔Confirm 的映射、以及命令分类器的复用，全部以下文 **§12 实现契约（已锁定）** 为准。本表保留为分级的概念描述。

## 5. 开关与生命周期

- REPL：`/auto on` / `/auto off`，**session 级**，**默认关闭**（安全默认，必须显式开启）。
- headless/CI：`--auto` 标志开启；无标志即关闭。
- 默认始终关闭：auto mode 永远是显式 opt-in，不存在「默认就自主」。
- kill switch：`/auto off` 与 Ctrl-C 在**下一个工具边界**生效，不打断进行中的工具调用（与 `/goal` 暂停语义一致）。

---

## 6. 安全要求（非功能，硬性）

1. **never-auto 清单与「不确定即确认」不可被关闭**——不能被配置项、不能被 `/goal`、尤其**不能被 agent 从工具结果/文件内容里读到的指令**改变（防 prompt-injection：agent 读到的不可信内容绝不能开启 auto mode、扩充允许清单或翻越 never-auto 边界）。
2. **允许清单的来源必须可信**：来自用户显式配置，不得由仓库内容/工具输出动态注入或扩充。
3. **审计**：每一次自动放行都发一条审计事件（工具、参数摘要、命中的放行规则），使一次无人值守跑完后用户能完整回看「自动批了什么、为什么」。这是信任的前提。
4. **不削弱 sandbox**：auto mode 只决定「批不批」，执行隔离仍由 `internal/sandbox` 负责；理想情况下自动放行的范围应是 sandbox 也能约束的范围。

---

## 7. 与 `/goal` 的关系

- `/goal` 的无人值守（hands-off）依赖 auto mode 开启。auto mode 未开启时，`/goal` 内核照常工作，但会停在 `ConfirmApprover` 的 y/N——这是**预期行为**，不是 bug。
- 汇合点：构造 `agent.Runner` 时注入哪个 `Approver`。goal 包不做任何 auto mode 特判。
- 因此两条线可独立开发、独立测试（`/goal` 用放行式 approver 替身测试，auto mode 用自己的策略测试），末端集成只是换实现。

---

## 8. 验收标准

1. 默认关闭时行为与现状完全一致：每个副作用调用都确认。
2. 开启后，工作区内的 edit_file 自动放行、不弹确认，并产生一条审计事件。
3. 开启后，命中允许清单的 `run_command`（如 `go test`）自动放行；未命中的命令（如 `curl …`）仍弹确认。
4. never-auto 清单：`rm -rf`、`git push`、`sudo`、作用于工作区外的路径——即使 auto mode 开启也始终确认。
5. prompt-injection：工具结果或文件内容里出现「自动批准一切」之类的指令，**不改变** auto mode 状态或允许清单。
6. kill switch：`/auto off`（或 Ctrl-C）在下一个工具边界停止自动放行，进行中的工具调用照常完成。
7. `/goal` 无人值守端到端仅在 auto mode 开启时成立；关闭时 `/goal` 停在确认（文档化的预期）。
8. fail-safe：无法分类的动作默认转人工确认。

---

## 9. 风险与开放问题（请开发重点评估）

1. **命令分类是最难、最危险的部分**：shell 命令含管道、`&&`、环境变量、子 shell、变量展开时极难安全判定。允许清单匹配必须保守——只匹配完全识别的简单命令，任何带 shell 元字符、可能逃逸的，一律转人工。这条做不好，auto mode 就是个安全漏洞。
2. **路径限定的鲁棒性**：「在 workspace root 内」要经得起符号链接、`..`、绝对路径、相对路径的规范化攻击。需要 canonicalize 后再判断。
3. **允许清单的存放与防污染**：放哪（项目配置）、如何防止被它要读取的仓库内容投毒。
4. **与 sandbox 的边界划分**：auto mode（批不批）与 sandbox（执行隔离）是互补的两层，确认两者协同、auto mode 不绕过 sandbox。
5. **headless/CI 的默认与退出行为**：CI 场景下的默认级别、以及与 `/goal` headless driver 的 exit code 协同。
6. **分级粒度**：Phase 1 是否只要「关 / 开」一个开关就够，还是需要「自动编辑但确认命令」这种中间档（建议 Phase 1 先一个开关 + 硬 never-auto 清单，中间档放 Phase 2）。

---

## 10. 分期建议

- **Phase 1（解锁 `/goal` hands-off）**：`AutoApprover` 实现 `Approver`；默认关闭；`/auto on|off` + headless 标志；策略 = 工作区内写自动放行 + `run_command` 经允许清单 + 硬编码 never-auto 清单 + 拒绝清单优先 + 不确定即确认；每次放行发审计事件；kill switch。
- **Phase 2**：更细的级别/配置、允许清单的配置 UX、更丰富的拒绝清单、可能的「批量确认/dry-run」模式。

---

## 11. 决策记录（ADR）

1. **drop-in Approver，不改 loop**：auto mode 是 `Approver` 的一个实现，调用路径不变。
   - **v0.2 实现修正**：审计为带 session 关联的可回看事件，loop 必须参与发射（关联 ID 只在 loop）。故 `permission.go` 的 `approve` 辅助新增一个可选接口 `AuditedApprover` 的分支（与 `SkillAnnouncer` 同模式），经 `r.emit` 发 `EventAutoApproved`。调用路径与 `Approver` 接口不变，仍是 drop-in；「loop 一行不改」放宽为「调用路径不改、仅 `approve` 辅助加关联发射」。详见 §12.3.2。
2. **默认关闭、显式 opt-in**：永不存在「默认自主」。
3. **never-auto 清单是硬规则、不可覆盖**（含不可被注入内容覆盖）。
4. **不确定即确认（fail-safe to human）**：分类不自信时默认人工，绝不默认自动。
5. **`run_command` 仅经允许清单自动放行，拒绝清单优先**。
6. **每次自动放行都审计**。
7. **auto mode 是 `/goal` 的上游依赖，汇合点仅在 `Runner.Approver` 注入**；两条线并行、互不阻塞。

## 12. 实现契约（v0.2，代码核对后锁定）

> 本节是对 §4/§5/§6 的**代码核对修订**，与上文冲突处以本节为准。所有判断都对到真实代码（文件:行），供实现直接照做，不必再自己推。决策已与需求方逐点签字（见末尾 ADR 增补）。

### 12.1 改变工作量判断的核心事实：Approver 的决策面比 §4 假设的小

loop 只对 `tools.HasSideEffectsFor(tool, input)==true` 的调用咨询 Approver（[internal/agent/loop.go:367](../internal/agent/loop.go#L367)）。对 `run_command`，`SideEffectsFor` 返回的正是 `sandbox.Classify(command).Decision == Confirm`（[internal/tools/shell/run_command.go:250](../internal/tools/shell/run_command.go#L250)）。由此三档命令的归宿：

- **`Allow` 档**（`go test`/`go build`/`go vet`/`go fmt`/`git status`/`git diff`/`ls`/`cat`/`grep`…，见 [sandbox/policy.go:94](../internal/sandbox/policy.go#L94) `DefaultPolicy`）→ `HasSideEffectsFor==false` → **今天就直接自动跑，永不咨询 Approver**。auto mode 对它们零作为。
- **`Block` 档**（`rm -rf /`、`git push --force origin main/master`、`bash -c`、`sudo`? 见下）→ 也不到 Approver；`run_command.Execute` 直接结构化拒绝（[run_command.go:142](../internal/tools/shell/run_command.go#L142)）。
- **`Confirm` 档**（`rm`/`mv`/`cp`/`curl`/`wget`/`ssh`/`sudo`/`git commit`/`git checkout`/`git reset`/`git push`/`npm`/`pip`/`docker` + **任何不认识的命令**，fail-safe 到 Confirm，[policy.go:201](../internal/sandbox/policy.go#L201)）→ **这是 Approver 在命令侧看到的全部**。

**关键推论**：Approver 看到的 `Confirm` 档命令集合，几乎**就是**文档 §4 的 never-auto 清单本身（rm/curl/sudo/git push…）。所以 auto mode 在命令侧没有「安全命令要放行」的缺口——安全命令早自动跑了。

**命令分类器无需重造**：§9.1 担心的保守命令分类，[internal/sandbox](../internal/sandbox) 已建好且满足全部要求——最长前缀+词边界匹配、不认识即 Confirm、硬编码 block 清单、`unquotedStructure` 剥引号内容防 commit message 误伤、`ContainsShellOperators` 拒 `| & ; > < $( ` \n`（[command_guard.go:13](../internal/sandbox/command_guard.go#L13)）。auto mode **复用 `sandbox.CommandPolicy`，不另起分类器**。

### 12.2 Phase 1 自动放行策略（锁定）

auto mode 的决策面 = `{edit_file, create_file, apply_patch, git_commit}`（静态副作用工具）+ `run_command` 的 Confirm 档。Phase 1 逐项裁定：

| 到达 Approver 的调用 | Phase 1 行为 | 依据 |
|---|---|---|
| `edit_file` / `create_file` | 目标路径在 workspace root 内 → **自动放行 + 审计**；否则人工 | 复用 `editFileInput.Path`/`createFileInput.Path` + `filepath.Join(rootAbs, Clean(Path))→Abs→workspace.IsSubPath`，与工具 Execute 内的检查逐字相同（[edit_file](../internal/tools/filesystem/edit_file.go)、[create_file.go:78](../internal/tools/filesystem/create_file.go#L78)） |
| `apply_patch` | **人工**（Phase 1 不自动放行；scope 决策见 §12.6） | 只有一个 `Patch` 字符串、无 `Path` 字段，一个 diff 可改多文件甚至带 `../`；静态不可判。**已核**：worker 的被引导主路径是 edit_file/create_file（见 §12.6），apply_patch 非头号用例的主编辑机制，故 Phase 1 默认留人工 |
| `git_commit` | **人工**（Phase 1 不自动放行） | **硬理由是 hook 执行**：`git_commit` shell out 到真 `git commit` 且不带 `--no-verify`（[commit.go:100](../internal/tools/git/commit.go#L100)），会触发 `.git/hooks/` 的 pre-commit/commit-msg/post-commit——那是任意 shell 代码。自动放行 commit = 自动放行任意 hook，**打破** edit_file 够格自动放行所依赖的「可逆+限工作区+有界爆炸半径」。本地 commit 本身（reset/reflog 可逆、不 push、不越界）风险并不高于 edit_file，所以「动历史」**不是**留人工的真因——hook 才是。自主循环的 checkpoint 需求**不靠放开通用 git_commit 满足**（任何一轮都能用任意 message 调它），而是 Phase 2 做成 goal 层、显式 opt-in 的里程碑提交（commit 语义由 goal engine 定） |
| `run_command`（任意 Confirm 档命令） | **人工**（Phase 1 一律不自动放行） | Confirm ≈ never-auto；安全命令已在 Allow 档自动跑。trusted run_command 允许清单留 Phase 2 |

**defense-in-depth（非唯一闸）**：auto mode 的路径检查只是「跳不跳人工」的前置闸；edit_file/create_file 的 Execute 各自独立再查一遍 `IsSubPath` 并拒绝越界写。即便前置闸被绕过，工具仍兜底。**但 `run_command` 没有等价的「写入限工作区」兜底**（`rm` 能删任何路径），这正是 Confirm 档命令 Phase 1 全留人工的硬理由。

**never-auto 的落地** = `sandbox` 硬编码 Block 清单 + Phase 1 把整个 Confirm 档留人工。无需新写一份 never-auto 清单。

### 12.3 开关 / 审计 / 注入契约

1. **单一 `AutoApprover` 包住人工 `ConfirmApprover`**，内含原子 `enabled`，**默认 false**。关闭时逐条委派人工 → 行为与今天逐字相同（直接满足验收 §8.1）。`/auto on|off` 翻这个 bool；`--auto` 旗标设初值。kill switch 自然在下一个工具边界生效（loop 每个工具调用前才咨询 Approver）。`/goal` 始终注入同一个 Approver，零特判。
2. **审计（实现时修正，见下）**：`EventAutoApproved` 由 **loop 发射**，不是 AutoApprover 自己发。原计划「AutoApprover 持 Emitter 自己 emit」在实现时被一处机制核查推翻：审计事件必须带 `SessionID`/`TurnID` 才能被**按 session 回看**（复看路径是 `SQLiteStore.SessionEvents(sessionID)`，`WHERE session_id=?`；[sqlite_store.go:393](../internal/session/sqlite_store.go#L393)），而这俩关联 ID **只有 loop 持有**（`r.emit` 统一盖 `At`/`SessionID`/`TurnID`，[loop.go:87-89](../internal/agent/loop.go#L87)）。AutoApprover 直接 emit 会产生**无 session 归属的孤儿事件**，回看不到——破坏 §6.3 的审计要求。
   - **改法（idiomatic，复用项目既有模式）**：新增**可选扩展接口** `agent.AuditedApprover{ Approver; ApproveAudited(tool,input)(approved bool, autoReason string) }`（[permission.go](../internal/agent/permission.go)）。`r.approve` 在 approver 实现该接口且**自动放行**（reason 非空）时，经 `r.emit` 发一条**带关联 ID** 的 `EventAutoApproved`。这与 loop 既有的 `SkillAnnouncer`/`SideEffectingInput` 可选接口同一路子——**调用路径（`HasSideEffectsFor && !r.approve`）不变**，仍是 drop-in Approver，只在 `permission.go` 的 `approve` 辅助里加了关联发射。AutoApprover 因此**不再持 Emitter**（更简）。人工确认/拒绝 → reason 空 → 不发审计（人已亲见）。
   - 事件经同一 `withEventStore` 持久化（除 token delta 外全存，[main.go:249](../cmd/codeagent/main.go#L249)）+ console 实时渲染，故**可持久回看 + 可实时看见**，§6.3 双满足。
3. **注入点 = `buildRunner`**（[cmd/codeagent/main.go:614](../cmd/codeagent/main.go#L614)），三入口共用的唯一缝。`/auto` 命令挂进 `repl.go` 的 `handleCommand` switch（[cmd/codeagent/repl.go:181](../cmd/codeagent/repl.go#L181)）。

### 12.4 污染向量（§6 的代码落地，硬约束）

`config.yaml` 按 **cwd 相对路径**加载（`app.LoadConfig("config.yaml")`，[main.go:49](../cmd/codeagent/main.go#L49)），且 `Workspace.Root` 默认 `.`（[config.go:249](../internal/app/config.go#L249)）——**config.yaml 通常就落在 agent 能写的工作区内**。因此：

- auto mode 的**开关与任何允许清单，绝不能来自 `config.yaml` 或任何仓库内文件**。可信来源只有：`--auto` 旗标、`/auto` REPL 命令、或工作区**外**的用户级配置。
- never-auto / Block 清单保持 **Go 硬编码**，永不读配置（现状已是）。
- agent 能通过工具输出/文件内容影响的，只有被判定的那个 `input` 本身（这正是要判的动作）；它**碰不到** `enabled` 与策略来源。这就是 prompt-injection 不能翻越边界的结构性保证。

**已知 gap（与现状一致，Phase 1 跟随）**：`workspace.IsSubPath` 是词法判断（`filepath.Rel` after `Clean`），**不解 symlink**（[workspace/path.go:8](../internal/workspace/path.go#L8)）。现有工具也不 `EvalSymlinks`，故 Phase 1 跟随现状；symlink 硬化（加 `EvalSymlinks`）作为 Phase 2 的独立可选项。

### 12.6 worker 编辑工具核查 + apply_patch 的 hands-off scope 决策

**核查（决定 Phase 1 算不算真 hands-off）**：worker 用哪个编辑工具？
- edit_file/create_file/apply_patch **均已注册**（[main.go:677-682](../cmd/codeagent/main.go#L677)）。
- `agentWorker` 走**全量 runner、零工具限制**（[engine.go:289](../internal/goal/engine.go#L289) `w.runner.RunTurn`），模型自选。
- `AgentSystemPrompt`（[prompt/system.go](../internal/prompt/system.go)）与 skills **无任何**「优先用 patch/diff」措辞；导向全在工具 Description——edit_file=「替换精确唯一文本」(单点修改的自然选择)、create_file=建新文件、apply_patch=应用 unified diff(多文件/结构性才用)。

**结论**：edit_file/create_file 是被引导的主编辑路径，头号用例「实现→测试→迭代」的编辑落在它们一侧 → §12.2 现状对头号用例**够 hands-off**。**残留风险（诚实标注）**：模型行为无法静态证明，某些模型对多 hunk 编辑会偏 apply_patch，每次都会停人工。

**Scope 决策（已签字：选 A）**：
- **A（已采纳）**：apply_patch Phase 1 留人工。头号用例不依赖它；apply_patch 的 diff 解析有真实边角（rename/delete/`/dev/null`/strip level/mode change），保守实现遇歧义本就 fall back 人工，边际 hands-off 收益小于「读 `+++ b/` 行」的直觉。完整自动化留 Phase 2（见 §12.7 backlog）。
- ~~B（更鲁棒，未采纳）~~：把 diff 路径提取提前进 Phase 1——解析 `+++ b/<path>` 逐个 `IsSubPath`，任一越界/歧义/binary/mode-change → 人工。无论模型选哪个编辑工具都 hands-off，代价是多一块解析+边角测试。**未采纳**，因 edit_file/create_file 已覆盖头号用例。
- **触发器（写给后续）**：若真实 `/goal` 跑动中观测到 worker 频繁用 apply_patch 而停在人工，即 Phase 2 把 B 的提取做掉的信号。

### 12.7 Phase 2 backlog（别让它们沉底）

- **apply_patch 自动放行**（若 §12.6 选 A，则在此完成 diff 路径提取）。
- **symlink 硬化**：`IsSubPath` 词法判断不解 symlink（[workspace/path.go:8](../internal/workspace/path.go#L8)）。**auto mode 下比手工审批更重要**——手工时人眼还能瞄到诡异路径，自动放行少了这道眼睛、且展示的是词法路径。Phase 2 加 `EvalSymlinks`，勿沉底。
- **goal 层 checkpoint 提交**：显式 opt-in 的里程碑 commit，替代「放开通用 git_commit」。
- **trusted run_command 允许清单**：用户级外部配置的少数 Confirm 档命令自动放行，仍受 Block/never-auto 压倒。

### 12.5 ADR 增补（v0.2，已签字）

8. **Approver 决策面 = Confirm 档 + 静态副作用工具**；`Allow` 档（test/build/format/git 只读）早已自动跑、不经 Approver——auto mode 不重复处理。
9. **Phase 1 命令侧一律人工**（Confirm≈never-auto；安全命令已自动跑）；文件自动放行**仅限 edit_file/create_file**。**git_commit 留人工的硬理由是 hook 执行**（shell out 无 `--no-verify`，触发任意 `.git/hooks/` shell），非「动历史」；checkpoint 需求由 Phase 2 goal 层 opt-in 提交满足。**apply_patch 留人工**（edit_file/create_file 是被引导主路径，§12.6；自动化留 Phase 1-B 或 Phase 2）。
10. **`config.yaml` 对 auto-mode 开关/允许清单不可信**（在工作区内、agent 可写）；开关只认 `--auto`/`/auto`/用户级外部配置，never-auto 恒 Go 硬编码。
11. **复用 `sandbox.CommandPolicy` 做命令判定，复用工具的 `*Input` 结构体 + `workspace.IsSubPath` 做路径判定**——不另起分类器（与项目复用 `model.IsRetryable`/`tools.HasSideEffectsFor` 同路子）。
